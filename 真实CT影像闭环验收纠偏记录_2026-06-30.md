# 真实 CT 影像闭环验收纠偏记录

更新时间：2026-06-30

## 纠偏结论

此前将“Project2 主平台到 Orchestrator、报告服务、EMR 服务的业务链路打通”表述为“可交付版本”，口径不准确。

当前必须区分：

- 业务链路打通：主平台接口、任务流转、报告审核发布、EMR 归档能够完成。
- 真实 CT 影像推理闭环：用户上传 CT 影像后，伪影模型真实读取影像并输出伪影结果，病灶模型真实读取同一影像并输出模型结果，报告与 EMR 基于该真实模型结果生成与归档。

只有第二项也通过，才能称为“真实 CT AI 工作链路可交付”。

## 已确认事实

1. `scripts/e2e_project2_real_business.py` 确实通过 Project2 的 `/api/ai/image/upload` 上传了 `testdata/headct/head_ct_positive_case.nii.gz`。
2. 该文件是项目测试 NIfTI 样本，不等同于真实临床 DICOM 序列验收数据。
3. Project2 的 AI 服务默认存在 mock 开关，真实验收必须确保 `ai.mock.enabled=false`。
4. Filter 服务当前登记了 InDuDoNet checkpoint，但代码中仍标注 InDuDoNet 尚未真正执行图像校正。
5. Lesion 服务存在 checkpoint fallback 机制，真实验收不能允许 fallback 路径通过。
6. 因此，之前的端到端结果只能证明“业务服务链路已衔接”，不能证明“真实 CT 影像 AI 推理闭环已达可交付”。

## 已完成纠正

已收紧 `scripts/e2e_project2_real_business.py` 的验收条件：

- 只接受 `projectUseStatus == ready_for_project_demo`。
- 必须 `workflowReady == true`。
- `aiImagingStatus.limitations` 必须为空。
- 病灶模型不得出现 `checkpointFallbackUsed == true`。

这意味着：

- 只要仍存在模型 fallback、mock、工作流未完全 ready、InDuDoNet 未执行、病灶输出能力不足等限制，真实业务 E2E 就应该失败。
- 后续不能再用宽松脚本结果声称“真实 CT AI 可交付”。

## 当前不可宣称可交付的点

1. 真实临床 DICOM series 上传与归一化仍需用实际样本验收。
2. 金属伪影模型 InDuDoNet 仍需完成可执行网络、投影域预处理、校正输出接入。
3. 病灶识别模型必须关闭 smoke/local fallback，并证明使用成熟 checkpoint 正常推理。
4. 医生端必须展示原始 CT、伪影 mask/校正结果、病灶结果、AI 报告、EMR 归档证据。
5. Playwright 截图验收必须覆盖真实上传、真实推理结果、报告发布、EMR 归档页面。

## 后续验收通过口径

只有同时满足以下条件，才允许标记为可交付：

1. 使用真实 CT 文件上传，而不是仅构造结构化结果。
2. Project2 上传接口保存实际文件，并将文件传给 Orchestrator。
3. Orchestrator 将同一影像传给 Filter 与 Lesion 服务。
4. Filter 输出真实伪影识别结果和可视化产物。
5. Lesion 输出真实模型推理结果，且无 checkpoint fallback。
6. Report 服务使用真实 AI 结果生成报告。
7. EMR 服务归档发布后的报告。
8. 前端医生端能查看完整结果。
9. 严格 E2E 和截图级验收均通过。
