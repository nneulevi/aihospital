## 一、实训项目概述

本阶段实训围绕“智慧云脑诊疗平台”中的影像识别相关需求展开，重点完成 Filter 子模块的设计、开发、验证与交付整理。该模块面向头部 CT 医学影像中的金属伪影识别场景，主要目标是从 CT 体数据中检测高亮金属区域及其可能造成的伪影影响，生成二值掩码、统计指标和可视化结果，为后续头部 CT 病灶识别、医生审核和 AI 智能检查报告生成提供影像质控依据。

在整体定位上，本阶段没有将 Filter 简单扩展成完整诊断系统，而是将其明确为“头部 CT 医学影像识别流程中的影像质控与金属伪影识别子模块”。该定位更加符合医疗 AI 系统的工程边界：Filter 负责读取影像、生成金属伪影 mask、输出结构化质控结果和可视化材料；最终医学判断仍应由后续识别模型和医生审核环节共同完成。

项目采用 Python 技术栈，综合使用 NumPy、SimpleITK、PySide6、VTK、Streamlit、PyTorch、FastAPI 和 PyInstaller 等工具。开发内容覆盖传统规则滤波、深度学习模型训练与推理、桌面端可视化、Web 演示界面、HTTP 服务化接口、数据处理脚本、模型产物管理和阶段性文档沉淀，整体工作具有较强的工程综合性。

## 二、本阶段主要工作内容

### 1. 需求分析与模块边界梳理

实训开始阶段，我首先阅读并整理了 CT 金属伪影检测相关文档，分析了金属植入物、骨性高亮区域、扫描噪声和条纹伪影在 CT 图像中的表现差异。结合项目目录中的说明文档和后续任务指导，我将模块职责拆分为影像读取、候选区域提取、掩码后处理、结果统计、可视化展示、模型推理和服务化输出几个部分。

在业务边界方面，我重点区分了“影像质控子模块”和“完整头部 CT 诊断平台”的差别。Filter 当前主要服务于金属伪影识别和 mask 生成，不直接承担脑出血、颅骨骨折、脑梗、中线移位等完整诊断任务，也不涉及挂号、问诊、电子病历和药房管理等业务。通过这一步梳理，后续开发目标更加集中，避免了功能范围失控。

### 2. 传统规则滤波流程实现

在算法基础部分，我完成并整理了基于 SimpleITK/NumPy 思路的金属伪影掩码生成流程。核心处理链路包括：

| 步骤 | 处理内容 | 作用 |
| --- | --- | --- |
| 二值化阈值滤波 | 按 HU 强度筛选高亮区域 | 初步定位疑似金属区域 |
| 梯度幅值约束 | 根据局部灰度变化保留强边缘 | 减少均匀高亮组织误检 |
| 形态学开运算 | 腐蚀后膨胀 | 去除孤立噪声点 |
| 形态学闭运算 | 膨胀后腐蚀 | 填补区域内部孔洞 |
| 连通域过滤 | 按最小体素数筛除小区域 | 保留较稳定的真实候选区域 |

对应实现主要体现在 `metal_artifact_mask_tool.py` 中。该工具支持内置 demo 体数据，也支持读取 `.npy`、`.nii`、`.nii.gz`、`.mha`、`.mhd` 等格式。处理完成后可输出二值掩码和统计 JSON，统计内容包括输入体数据尺寸、强度范围、每一步剩余体素数量、连通域数量、最大连通域规模和所用参数。通过这些指标，可以判断当前阈值、梯度阈值和形态学半径是否过松或过严，为调参提供依据。

### 3. 可视化界面与交互工具开发

为了让算法结果更直观，本阶段实现了两类可视化工具。一类是基于 Streamlit 的轻量 Web 演示界面 `metal_artifact_mask_app.py`，支持内置 demo 或上传体数据，能够在页面中调节阈值、梯度阈值、开闭运算半径和最小连通域面积，并展示原始图像、阈值阶段、梯度阶段、形态学阶段和最终掩码结果。该界面适合快速演示算法流程和参数变化效果。

另一类是基于 PySide6 与 VTK 的桌面端工具 `metal_artifact_mask_desktop.py`。桌面端工作量更大，功能也更接近实际标注工具，主要包括：

- 支持加载内置 demo、本地 `.npy/.nii/.nii.gz/.mha/.mhd` 文件以及 DICOM 序列文件夹。
- 支持三平面视图，包括 axial、coronal、sagittal 三个方向的切片浏览。
- 支持 CT 原图与 mask 红色叠加显示，便于观察伪影位置。
- 支持滑块切片选择、窗宽窗位显示范围、参数交互调节和异步加载。
- 支持使用 VTK Marching Cubes 进行 3D 掩码表面重建，可旋转、缩放和平移观察。
- 支持将最终掩码保存为 `.npy`，在 SimpleITK 模板可用时保存为医学影像格式。

这一部分让我实际接触了医学体数据的三维显示逻辑，也加深了对 SimpleITK 图像空间信息、NumPy 数组维度顺序、VTK 数据转换和 GUI 异步任务处理的理解。

### 4. 数据处理与样本产物整理

项目中包含 CQ500 颅脑 CT 数据集相关说明文档和数据字段解释。我在实训过程中对数据组织方式、DICOM Study/Series、NIfTI 转换和掩码输出目录进行了整理。`Mask_mark/output` 目录中已经形成了按病例和序列划分的输出结果，每个序列通常包含：

- `ct_volume.nii.gz`：转换后的 CT 体数据；
- `metal_mask.nii.gz`：生成的金属伪影掩码；
- `stats.json`：处理参数和体素统计结果。

根据 `manifest.json`，当前批处理记录覆盖 25 条序列样本，包含不同层厚、不同序列名称和不同体数据深度的 CT 输入。部分样本的体数据尺寸达到 `288 x 512 x 512`、`313 x 512 x 512`、`325 x 512 x 512` 等规模，说明本阶段并不是只处理小型 toy data，而是围绕真实医学影像体数据进行了流程验证。

通过这些数据处理产物，我进一步理解了真实医学影像工程中“数据读取、空间信息保留、批处理输出、结果统计、文件命名规范”的重要性。对于后续训练集构建和医生审核回流，这些结构化产物也是基础。

### 5. 深度学习模型方案补充

除规则滤波外，本阶段还补充了 2D U-Net 与 3D U-Net 两条深度学习路线，用于探索自动分割能力。

2D U-Net 方案位于 `model/UNet2D方案`，主要将 3D CT 体数据拆分为 axial 方向二维切片进行训练和推理。该方案训练速度较快、显存占用较低，适合作为 baseline 或低资源 fallback。训练脚本中加入了切片缓存、正负样本处理、空切片抽样、`pos_weight` 统计、CUDA DataLoader 优化等工程处理。当前 smoke 训练记录显示，2D 方案处理了 10 条记录、517 张切片，其中训练集 414 张、验证集 103 张，正样本训练切片 365 张，已经形成 `best_unet2d_metal.pt`、`last_unet2d_metal.pt` 和 `metrics.json` 等产物。

3D U-Net 方案位于 `model` 目录下，面向体素级分割任务，能够利用层间空间上下文。当前 smoke 训练记录中，数据按 20 条训练、5 条验证拆分，patch size 为 `16 x 64 x 64`，并保存了 `best_unet3d_metal.pt`、`last_unet3d_metal.pt` 和指标文件。虽然当前训练仍属于小规模验证性质，不能直接视为生产级模型效果，但它证明了从数据集、模型结构、训练脚本到权重输出的基本链路已经打通。

通过同时保留规则滤波、2D U-Net 和 3D U-Net，本阶段形成了较合理的算法分层：规则法适合作为快速 baseline 和无模型 fallback；2D U-Net 适合快速推理和低显存环境；3D U-Net 适合后续追求更稳定的体数据分割效果。

在后续模型升级方向上，项目还可以进一步参考 nnU-Net 和 MONAI 等医学影像开源框架。nnU-Net 的优势在于能够根据数据集特征自动配置预处理、网络结构、patch size、训练策略和后处理流程，适合在标注数据逐步积累后作为更规范的分割基线；MONAI 则提供了医学影像数据加载、空间变换、数据增强、loss、metric、滑窗推理和模型组件等完整工具链，有助于减少自研训练代码中的重复工作，并提升实验复现性。当前阶段我主要完成了自研 2D/3D U-Net 链路和 smoke 级验证，后续可以将 nnU-Net/MONAI 作为对照方案或重构方向，用于提高模型训练规范性和工程可维护性。

### 6. FastAPI 服务化封装

为了让算法能力能够被平台调用，本阶段还实现了 FastAPI 微服务入口。当前 `Fastapi/CTDetectionServer.py` 和 `Fastapi/CTDetectionServer_unet2d.py` 分别对应 3D 与 2D 模型服务化思路，主要能力包括：

- `GET /health`：返回服务状态、模型状态、存储目录和推理参数；
- `POST /predict-ct-artifact`：上传 `.nii` 或 `.nii.gz` CT 文件并生成 mask；
- `GET /results/{mask_filename}`：下载生成的掩码结果；
- `/ui`：提供基础前端页面入口；
- 模型加载失败时服务仍可启动，并通过 degraded 状态说明原因；
- 支持上传目录、结果目录、模型权重路径、设备、patch size、overlap 和 threshold 等配置项。

该部分的意义在于把算法脚本推进到“可被其他系统调用”的形态。虽然当前接口还需要进一步补充异步任务、统一错误码、结构化 result.json、模型版本和日志审计等内容，但已有服务已经具备微服务雏形，后续可以继续演进为平台级接口。

## 四、遇到的问题与解决思路

### 1. 医学影像格式和空间信息处理复杂

CT 数据并不是普通二维图片，NIfTI、MHA/MHD、DICOM 序列都包含 spacing、origin、direction 等空间信息。如果处理掩码时丢失这些信息，后续叠加显示和医学软件读取就可能出现位置偏差。为解决这一问题，我在保存 mask 时尽量复制原始 SimpleITK 图像的信息，保证输出掩码与输入 CT 在空间上保持一致。

### 2. 金属伪影与骨骼高亮区域容易混淆

CT 图像中骨骼、金属和部分噪声都可能表现为高亮区域，单纯依赖阈值容易产生误检。因此规则法中加入了梯度幅值约束、形态学开闭运算和连通域过滤，用多个步骤逐步减少噪声。后续仍需要结合真实标注数据和深度学习模型进一步提升准确性。

### 3. 三维数据可视化性能压力较大

在桌面端中，三平面显示和 VTK 3D 重建都需要处理较大的体数据。如果所有操作都在主线程执行，界面容易卡顿。因此界面中引入了异步加载、后台工作线程和进度提示，并在切片显示时只对当前切片进行归一化和叠加绘制，减少交互延迟。

### 4. 深度学习训练样本和指标仍需积累

当前 2D/3D U-Net 已经完成 smoke 级训练和推理链路，但训练规模、标注质量、验证指标和泛化能力仍不足以支撑正式医疗应用。后续需要扩大标注数据、区分训练/验证/测试集、记录模型版本、固定阈值策略，并加入更完整的 Dice、IoU、Recall、Precision 等指标对比。

### 5. 服务化接口还需平台级规范

当前 FastAPI 服务已经可以上传 NIfTI 并返回 mask，但正式接入平台还需要任务 ID、异步推理、统一错误码、result.json 落盘、模型版本、推理耗时、严重程度、报告建议和审计日志等字段。对此，我在后续工作指导中已经整理了接口演进方向，建议优先补齐 `/api/ct-artifact/...` 风格的稳定接口契约。

## 六、后续改进计划

结合当前项目状态，我认为后续应按以下优先级继续推进：

1. 统一 2D 与 3D 服务输出字段，补充 `artifact_detected`、`artifact_ratio`、`severity`、`affected_slices`、`model_version`、`elapsed_ms` 等结构化字段。
2. 增加 `result.json` 落盘机制，将 mask、统计信息、模型信息和报告建议放入同一结果目录。
3. 增加三平面预览图导出能力，输出 `preview_axial.png`、`preview_coronal.png`、`preview_sagittal.png`，方便医生端和报告端使用。
4. 将服务接口整理为 `/api/ct-artifact/health`、`/api/ct-artifact/detect`、`/api/ct-artifact/results/{request_id}` 等更稳定的形式。
5. 增加异步任务机制，上传后先返回 `task_id`，后台推理完成后再查询结果，避免大体数据推理阻塞 HTTP 请求。
6. 扩展 DICOM zip 或 DICOM Series 输入，保存 StudyInstanceUID、SeriesInstanceUID 等元数据。
7. 建立医生审核闭环，支持医生修正 mask、保存 `review.json`，并将确认样本回流训练集。
8. 增加 smoke test 和接口测试，保证 CLI、GUI、模型推理和 FastAPI 服务在基础场景下可重复运行。
9. 引入 nnU-Net 或 MONAI 作为医学影像分割实验框架，对现有自研 2D/3D U-Net 方案进行基线对照、训练流程规范化和滑窗推理优化。
10. 建设轻量级 RAG 知识库，将项目文档、接口说明、训练记录、模型指标、参数解释和医学影像质控规范纳入检索范围，辅助后续开发人员快速查询背景资料、生成报告建议和定位模型版本差异。

当前 Filter 已经具备“可演示原型 / AI 推理微服务雏形”的能力，能够体现为头部 CT 影像质控与金属伪影识别子模块的核心基础。后续如果继续补齐接口契约、异步任务、结构化结果、模型版本、预览图、医生审核和测试材料，就可以逐步从 demo 阶段推进到 MVP 级可交付子模块。

从更长期的演进看，Filter 还可以在两个方向继续增强：一是引入 nnU-Net、MONAI 等成熟医学影像框架，提高模型训练、评估和部署流程的标准化程度；二是建设面向项目内部资料的 RAG 知识库，让系统能够检索数据集说明、接口文档、训练日志、模型指标和医学质控说明，为开发调试、医生审核和 AI 检查报告文本生成提供辅助。

通过本次实训，我不仅完成了具体功能开发，也更加理解了真实软件项目中需求分析、架构边界、算法验证、用户界面、服务接口和交付文档之间的关系。这些经验对我后续参与医疗影像、人工智能和工程化系统开发都有直接帮助。
