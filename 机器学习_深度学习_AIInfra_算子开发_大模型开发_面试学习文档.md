## 使用这份文档的方式：从 0 到可面试

如果你是 0 基础，不要一开始就试图记住所有名词。机器学习、大模型和 AI Infra 的知识看起来很多，但它们其实围绕同一件事展开：**让计算机从数据中学到规律，并把这个规律稳定、快速、可控地服务给真实用户**。

可以把整个学习过程理解成 6 层：

1. **数学层**：回答“模型为什么能学”。例如矩阵乘、概率、梯度、优化。
2. **模型层**：回答“模型长什么样”。例如线性模型、CNN、Transformer。
3. **训练层**：回答“模型怎么从数据中变好”。例如 loss、反向传播、优化器、数据划分。
4. **工程层**：回答“模型怎么在代码里跑起来”。例如 PyTorch、Dataset、checkpoint、debug。
5. **系统层**：回答“模型怎么服务真实用户”。例如推理服务、分布式训练、监控、缓存、限流。
6. **性能层**：回答“模型怎么跑得更快更省”。例如 CUDA、Triton、算子融合、NCCL、量化。

面试不是要求你“从头发明深度学习”，而是看你能否沿着这 6 层把问题讲清楚。比如面试官问“为什么你的推理服务慢”，一个 0 基础回答可能只会说“模型太大”。可面试的回答会拆开看：输入 token 是否太长、batching 是否合理、GPU 是否空等、KV cache 是否占满、CPU tokenization 是否成为瓶颈、网络和下游服务是否拖慢。这个拆解能力，就是从 0 到可面试的核心。

每学一个概念，都按四个问题理解：

- **它解决什么问题**：没有它会发生什么。
- **它的输入输出是什么**：数据从哪里来，变成什么。
- **它的核心逻辑是什么**：中间做了哪些计算或决策。
- **面试怎么讲**：用 30 秒说清楚，不陷入背定义。

例如学习 Attention：

- 它解决的问题：让模型在处理一个 token 时能关注上下文中相关 token。
- 输入输出：输入 Q、K、V 三组向量，输出融合上下文信息的新向量。
- 核心逻辑：Q 和 K 做相似度，softmax 变成权重，再对 V 加权求和。
- 面试表达：Attention 是一种内容相关的加权汇聚机制，复杂度通常随序列长度平方增长，所以长上下文推理要重点优化。

这份文档后面每章都会尽量按这个逻辑展开。

## 0. 面试时真正考察什么

面试不是背百科，而是看你能不能把一个 AI 系统从数据、模型、训练、算子、部署、监控、成本和风险串起来。

零基础先建立一个整体图景：AI 面试的本质不是考“你记住了多少模型名”，而是考你是否理解一个 AI 项目从开始到结束会经历什么。一个完整项目通常是这样的：

```text
业务问题 -> 数据 -> 特征/预处理 -> 模型 -> 训练 -> 评估 -> 部署 -> 监控 -> 反馈迭代
```

比如“头部 CT 辅助诊断”不是一句“用深度学习识别病灶”就结束了。你要先知道 CT 数据来自哪里，标注是谁做的，训练集和测试集有没有同一个病人泄漏；再知道模型是做分类、检测还是分割；然后知道用什么指标评估，错一个阳性和错一个阴性的代价是否一样；最后还要知道模型结果如何进入医生审核流程，出错时怎么兜底。这就是面试官真正想听的“框架感”。

### 0.1 导师面试关注点

导师常看：

- 基础是否稳：概率统计、线性代数、优化、深度学习基本概念。
- 是否有研究意识：能否把工程问题抽象成可验证假设。
- 是否能读论文：能否讲清论文解决了什么问题、方法假设是什么、实验是否充分。
- 是否能动手：是否训练过模型、调过数据、复现实验、分析过失败样例。
- 是否能长期推进：是否能把模糊问题拆成阶段性目标。

常见追问：

- 你做过的 AI 项目中，数据如何清洗？标签质量如何保证？
- 为什么选这个模型？有没有和更简单 baseline 比较？
- 你怎么判断模型真的有用，而不是只在测试集上好看？
- 如果部署到真实业务场景，延迟、吞吐、漂移、错误兜底怎么做？
- 你未来想做什么方向？为什么这个方向值得研究？

### 0.2 工作面试关注点

算法岗常看：

- ML/DL 基础、损失函数、优化器、正则化、模型结构、实验分析。
- 能不能从业务目标定义指标。
- 能不能处理数据脏、样本偏、线上线下不一致。

AI Infra 岗常看：

- 训练系统：分布式并行、通信、显存、checkpoint、容错。
- 推理系统：batching、KV cache、量化、流式输出、服务治理。
- 工程能力：API、队列、缓存、监控、压测、成本。

算子开发岗常看：

- GPU 架构、CUDA 编程模型、访存层级、线程组织。
- 性能分析：roofline、occupancy、memory bandwidth、算术强度。
- Kernel 优化：coalescing、shared memory、tiling、warp primitive、fusion。
- 框架集成：PyTorch extension、Triton、自定义 autograd、benchmark。

大模型开发岗常看：

- Transformer、Attention、位置编码、预训练和 SFT/RLHF/DPO。
- RAG、Agent、工具调用、长上下文、结构化输出。
- 推理优化：KV cache、paged attention、speculative decoding、量化、并行。
- 评测与安全：幻觉、偏见、鲁棒性、数据泄漏、线上反馈闭环。

## 1. 数学和计算机基础

这一章不要按“数学课”来学，而要按“AI 里为什么需要它”来学。数学不是为了推公式而推公式，它是帮你回答三个问题：

- 数据怎么表示：向量、矩阵、张量。
- 模型怎么计算：矩阵乘、非线性函数、概率输出。
- 参数怎么更新：loss、梯度、优化器。

计算机基础则回答另外三个问题：

- 为什么同一个模型在不同硬件上速度差很多。
- 为什么显存、内存、磁盘、网络会成为瓶颈。
- 为什么工程系统需要缓存、队列、并发和监控。

从 0 开始可以先记住一句话：**机器学习中的绝大多数计算，最后都会落到大量矩阵和张量运算；AI Infra 的绝大多数性能问题，最后都会落到计算、内存和通信的平衡。**

### 1.1 线性代数

必须掌握：

- 向量、矩阵、张量的形状和广播规则。
- 矩阵乘法：`(B, M, K) @ (B, K, N) -> (B, M, N)`。
- 点积、范数、余弦相似度。
- 特征值、特征向量、SVD 的直觉。
- 正交、投影、低秩近似。

这些东西分别是什么、怎么用：

- **向量是什么**：向量是一排数字，例如 `[年龄, 血压, 血糖]`，也可以是一段文本或图片经过模型后得到的 embedding。实际使用时，模型不会理解“年龄”这个词，只会处理向量里的数值。
- **矩阵是什么**：矩阵是二维数字表。一个 batch 的样本常表示为矩阵，例如 32 个样本、每个样本 768 维，就是 `[32, 768]`。Linear 层输入通常就是矩阵。
- **张量是什么**：张量是更高维的数组。图像常见 `[B, C, H, W]`，文本常见 `[B, T, H]`，分别表示 batch、序列长度、隐藏维度等。
- **广播规则怎么做**：当两个 tensor 形状不同但某些维度为 1 时，框架会“虚拟扩展”它们。例如 `[B, H] + [H]` 会把 `[H]` 当作每个样本共享的 bias 加上去。
- **矩阵乘怎么做**：`A[M,K] @ B[K,N]` 的结果是 `C[M,N]`，其中 `C[i,j]` 是 A 第 i 行和 B 第 j 列的点积。神经网络里 Linear、Attention、FFN 都大量使用它。
- **点积是什么**：两个向量逐元素相乘再求和。点积大，通常表示方向更相似或匹配程度更高。Attention 中 `QK^T` 本质就是大量点积。
- **范数是什么**：衡量向量大小，常见 L2 范数是平方和开根号。正则化、梯度裁剪、向量归一化都会用到。
- **余弦相似度是什么**：衡量两个向量方向是否接近，忽略长度影响。RAG 检索、embedding 匹配经常用它。
- **特征值/特征向量是什么**：矩阵作用在某些特殊方向上时，只改变长度不改变方向，这些方向是特征向量，缩放倍数是特征值。理解 PCA 时有用。
- **SVD 是什么**：把一个矩阵拆成几个更容易理解的矩阵，用于低秩近似、压缩和去噪。LoRA 的低秩思想可以借它建立直觉。
- **低秩近似怎么用**：用更小的矩阵乘积近似大矩阵，减少参数和计算。LoRA 就是在大模型权重旁边学习低秩增量。

面试表达：

- “深度学习中的 Linear 层本质是矩阵乘加；Attention 的核心也是 QK^T 和 softmax(V) 的组合。”
- “Embedding 检索常用余弦相似度或内积，本质上是在高维语义空间中做近邻搜索。”
- “低秩近似可以解释 LoRA：不直接更新完整大矩阵，而是学习两个小矩阵的乘积。”

典型问题：

- 为什么 PCA 可以降维？
- SVD 和特征值分解有什么区别？
- 为什么 Transformer 中矩阵乘性能很关键？

典型问题答案：

**为什么 PCA 可以降维？** PCA 的目标是找到一组新的坐标轴，让数据在前几个轴上的方差最大。方差大说明这个方向保留了更多变化信息，方差小的方向往往包含较少信息或噪声。降维时保留前几个主成分，就能在尽量少损失信息的情况下把高维数据压到低维。它常用于可视化、去噪和压缩，但深度学习里更多用 embedding 或神经网络自动学习表示。

**SVD 和特征值分解有什么区别？** 特征值分解通常要求矩阵是方阵，表示一个线性变换在某些方向上只缩放不改变方向；SVD 可以用于任意矩阵，把矩阵分解成 `U Σ V^T`，可以理解成“先旋转、再缩放、再旋转”。SVD 更通用，常用于低秩近似、矩阵压缩和推荐系统。

**为什么 Transformer 中矩阵乘性能很关键？** Transformer 的 Q/K/V 投影、attention score、FFN 都是大规模矩阵乘。矩阵乘占据大量 FLOPs，也决定 GPU Tensor Core 能否充分利用。所以大模型训练和推理优化往往先看 matmul、attention 和相关算子的吞吐。

从 0 理解：

向量可以理解成“一组数字”。例如一个病人的年龄、血压、血糖可以组成一个向量；一张图片的像素也可以拉平成向量。矩阵可以理解成“很多向量排在一起”，比如一个 batch 里有 32 张图片，每张图片经过模型后变成 768 维特征，那么这个 batch 就可以表示成 `[32, 768]` 的矩阵。张量则是更高维的矩阵，例如图片 batch 常见形状是 `[B, C, H, W]`，分别表示 batch 大小、通道数、高度、宽度。

为什么矩阵乘重要？因为神经网络的 Linear 层本质就是：

```text
输出 = 输入矩阵 * 权重矩阵 + 偏置
```

Transformer 中的 Q、K、V 投影、Attention 的 `QK^T`、FFN 的两层 Linear，也都主要是矩阵乘。也就是说，模型能力看起来很“智能”，底层却大量依赖矩阵乘能否快速完成。因此算子开发、CUDA、Tensor Core、cuBLAS 都围绕矩阵乘性能展开。

面试时可以这样讲：

“我理解线性代数不是孤立的数学知识，而是深度学习的计算语言。输入、权重、激活都可以表示成张量，模型前向过程主要由矩阵乘、归一化和非线性函数组成。Transformer 的性能瓶颈也经常来自大规模矩阵乘和 attention 相关计算。”

### 1.2 概率统计

必须掌握：

- 随机变量、分布、期望、方差。
- 最大似然估计和交叉熵。
- 条件概率、贝叶斯公式。
- 偏差和方差。
- 置信区间、显著性、A/B 测试基本概念。

这些概念分别是什么、怎么用：

- **随机变量是什么**：结果不固定的变量。例如“某张 CT 是否异常”可以看成 0/1 随机变量，模型输出的是它为 1 的概率。
- **分布是什么**：描述随机变量各种取值的概率。分类模型输出的 softmax 就是一个类别概率分布。
- **期望是什么**：随机变量的平均结果，可以理解成长期平均值。loss 通常是在数据分布上的期望风险，训练时用 batch 平均近似。
- **方差是什么**：衡量波动大小。模型结果方差大，说明对数据扰动敏感，可能泛化不稳定。
- **最大似然估计怎么做**：选参数让训练数据出现的概率最大。分类里等价于让正确标签概率最大，实际优化交叉熵。
- **条件概率是什么**：在某个条件成立时事件发生的概率，例如 `P(疾病 | 影像特征)`。分类模型本质上在估计条件概率。
- **贝叶斯公式怎么用**：把先验概率和证据结合起来更新判断。医学场景中同样影像征象在不同年龄、病史下意义可能不同。
- **偏差和方差怎么判断**：训练和验证都差，偏差高；训练好验证差，方差高。调模型时要先判断是哪种问题。
- **置信区间是什么**：指标不是一个绝对确定数字，样本量有限时会有不确定范围。小测试集上 1% 提升未必可靠。
- **A/B 测试怎么做**：把线上流量随机分给 A/B 两个版本，对比业务指标和风险指标，判断新模型是否真的更好。

面试表达：

- “分类任务的交叉熵可以看成最大化正确标签的对数似然。”
- “过拟合通常表现为训练误差下降但验证误差上升，本质上是模型对训练集噪声也进行了拟合。”
- “线上评估不仅看平均指标，还要看分桶指标，因为总体提升可能掩盖某些人群或场景的退化。”

高频问题答案：

**最大似然估计是什么？** 最大似然估计是在假设模型形式已知的情况下，寻找一组参数，让已观察到的数据出现的概率最大。分类模型中，正确类别概率越高，似然越大；取负对数后就得到常见的交叉熵 loss。

**交叉熵为什么适合分类？** 分类任务希望模型给正确类别更高概率。交叉熵会对“正确类别概率很低”的情况给出很大惩罚，比 MSE 更直接地优化概率分布，也和最大似然训练一致。

**偏差和方差是什么？** 偏差高表示模型太简单，系统性学不到规律，容易欠拟合；方差高表示模型太依赖训练数据细节，容易过拟合。调模型时常在偏差和方差之间平衡。

从 0 理解：

概率是用来表达“不确定性”的。模型做分类时，通常不是只说“这一定是猫”，而是输出每一类的概率，比如猫 0.8、狗 0.15、车 0.05。训练时，我们希望正确类别的概率越来越高，错误类别的概率越来越低。交叉熵就是衡量“模型预测分布”和“真实标签分布”差距的一种方式。

统计则帮助你判断结果是否可靠。比如一个模型在 100 张图上准确率 95%，另一个在 10 万张图上准确率 93%，不能简单说前者更好，因为样本量、数据分布、类别难度都不同。真实面试中，能主动提“样本量、置信度、分桶评估、数据偏差”，会比只背 accuracy 更有说服力。

常见逻辑：

```text
模型输出概率 -> loss 衡量预测和真实差距 -> 优化器让 loss 下降 -> 指标评估模型是否有业务价值
```

### 1.3 优化

必须掌握：

- 梯度下降、SGD、Momentum、Adam、AdamW。
- 学习率、warmup、cosine decay。
- 局部最优、鞍点、梯度消失、梯度爆炸。
- L1/L2 正则、weight decay、dropout、early stopping。

这些概念分别是什么、怎么做：

- **梯度下降是什么**：沿着让 loss 下降最快的反方向更新参数。实际训练中每个 batch 估计一次梯度。
- **SGD 怎么做**：用一个 mini-batch 的梯度更新参数，简单、稳定，但可能震荡或收敛慢。
- **Momentum 怎么做**：保留历史更新方向，相当于给梯度下降加惯性，减少来回震荡。
- **Adam 怎么做**：同时维护梯度的一阶矩和二阶矩，为不同参数自适应调整学习率，常用于深度学习默认 baseline。
- **AdamW 怎么做**：在 Adam 基础上把 weight decay 解耦，Transformer 和大模型训练中非常常见。
- **学习率是什么**：每次参数更新的步子大小。太大容易发散，太小训练慢。
- **warmup 怎么做**：训练开始先用小学习率，再逐渐升到目标学习率，避免初期不稳定。
- **cosine decay 怎么做**：学习率按余弦曲线逐渐下降，让训练后期更细致地收敛。
- **局部最优和鞍点是什么**：局部最优是附近都更差的位置；鞍点是某些方向升、某些方向降的位置。深度网络更常遇到鞍点和平台区。
- **梯度消失/爆炸是什么**：反向传播中梯度过小导致前层学不动，或过大导致参数剧烈变化。用残差、归一化、合适初始化、梯度裁剪处理。
- **L1/L2 正则是什么**：在 loss 中惩罚参数过大。L1 倾向稀疏，L2 倾向平滑小权重。
- **early stopping 怎么做**：验证集长期不提升就停止，并恢复最佳 checkpoint，防止继续过拟合。

AdamW 重点：

- Adam 把一阶矩和二阶矩用于自适应学习率。
- AdamW 将 weight decay 从梯度更新中解耦，常用于 Transformer 训练。
- 面试中不要只说“AdamW 更好”，要说它解决了 Adam 中 L2 正则和自适应学习率耦合的问题。

怎么做：

1. 先选一个稳定 baseline，例如 AdamW + 合理学习率。
2. 观察训练 loss 和验证指标：两者都差是欠拟合，训练好验证差是过拟合。
3. 如果 loss 震荡或 NaN，先降学习率、检查数据和梯度。
4. 如果训练很慢，可以加 warmup、调 batch size、用 cosine decay。
5. 如果验证集变差，考虑 weight decay、dropout、数据增强或 early stopping。

高频问题答案：

**SGD、Momentum、Adam 区别是什么？** SGD 每次按当前梯度更新；Momentum 会累积历史方向，减少震荡；Adam 同时估计一阶矩和二阶矩，为不同参数自适应调整步长。Adam 收敛快，SGD 有时泛化更好，具体要看任务。

**学习率怎么调？** 学习率过大 loss 震荡或发散，过小收敛慢。常用做法是先用经验值或 learning rate finder 找范围，再配 warmup 和 decay。大 batch 训练通常需要相应调整学习率。

**梯度爆炸怎么办？** 先确认是否有 NaN/Inf，再降低学习率、使用 gradient clipping、检查初始化和 loss。RNN、长序列和混合精度训练中尤其要注意。

从 0 理解：

训练模型可以类比“调旋钮”。模型有很多参数，这些参数一开始是随机的，所以预测很差。loss 告诉我们“现在错得有多离谱”，梯度告诉我们“每个参数往哪个方向调会让错误变小”。优化器就是根据梯度调整参数的方法。

SGD 像是每次沿着当前坡度走一步；Momentum 会记住之前的方向，让更新更平滑；Adam 会给不同参数自适应调整步子大小；AdamW 在大模型训练中常用，因为它把权重衰减处理得更稳定。

学习率是最重要的超参数之一。学习率太大，loss 可能震荡甚至 NaN；学习率太小，训练很慢甚至卡住。warmup 的直觉是：训练刚开始参数还很乱，先用小步走稳，再逐渐加大学习率。

### 1.4 计算机系统基础

必须掌握：

- CPU cache、内存局部性、SIMD。
- 进程、线程、锁、异步 IO。
- 网络基础：HTTP、RPC、负载均衡、超时重试。
- Linux 常用命令、日志定位、资源监控。
- 数据结构：数组、哈希表、堆、队列、图。

这些概念在 AI 工程里怎么用：

- **CPU cache 是什么**：CPU 访问内存前的小型高速缓存。数据连续访问更容易命中 cache，所以数组顺序访问通常比随机访问快。
- **内存局部性是什么**：程序如果反复访问相邻或近期访问过的数据，就更快。数据预处理、tokenization、向量检索都受它影响。
- **SIMD 是什么**：一条指令处理多个数据。CPU 上的向量化、矩阵库、图像预处理会用到。
- **进程和线程是什么**：进程资源隔离更强，线程共享进程内存。PyTorch DataLoader 多 worker、模型服务多进程都涉及它。
- **锁是什么**：保护共享数据不被并发写坏。锁用多了会降低并发，AI 服务中队列、缓存、日志都要注意。
- **异步 IO 是什么**：等待网络或磁盘时不阻塞主线程。流式推理、上传文件、调用下游服务常用。
- **HTTP/RPC 怎么用**：HTTP 面向通用 Web API；RPC 更偏服务间调用。模型服务、业务服务、网关经常通过它们通信。
- **负载均衡是什么**：把请求分发到多个 worker 或机器，避免单点过载。
- **超时重试怎么做**：每个下游调用设置 timeout；只对幂等或安全请求重试；避免无限重试放大故障。
- **队列怎么用**：推理请求排队、异步任务、削峰填谷都依赖队列。队列长度也是重要监控指标。

AI Infra 面试经常把模型问题问成系统问题：

- 为什么推理服务 GPU 利用率低？
- 为什么 batch size 加大后延迟变差？
- 为什么 CPU tokenization 成为瓶颈？
- 为什么多机训练带宽不够？

高频问题答案：

**为什么推理服务 GPU 利用率低？** 可能是请求太少、batching 不充分、CPU 预处理慢、tokenization 慢、数据拷贝多、模型 worker 排队策略不合理。排查时先看 GPU utilization、CPU 占用、请求队列、batch size、输入输出 token 长度。

**为什么 batch size 加大后延迟变差？** batch 变大通常提高吞吐，但单个请求要等 batch 凑齐，且一次计算更重，所以 P95/P99 延迟可能升高。线上要在吞吐和延迟 SLO 之间折中。

**为什么多机训练带宽不够？** 多机训练需要同步梯度或参数分片，通信量巨大。如果网络拓扑、NCCL 配置、bucket size 或 overlap 不合理，GPU 会等通信。排查要看通信时间占比和链路带宽。

从 0 理解：

AI 工程不是只有模型，还要让数据在 CPU、内存、磁盘、GPU、网络之间流动。速度慢通常不是“代码整体慢”，而是某个环节拖住了整个流水线。

例如训练时 GPU 很贵，但如果 DataLoader 从磁盘读图太慢，GPU 就会空等；推理时模型在 GPU 上很快，但如果 CPU tokenization 很慢，用户仍然会觉得慢；多机训练时每张卡本地算得很快，但梯度同步要跨机器通信，网络带宽就可能成为瓶颈。

面试中可以把系统问题拆成：

```text
数据读取 -> CPU 预处理 -> CPU/GPU 拷贝 -> GPU 计算 -> 通信/返回 -> 日志监控
```

这样排查会比笼统地说“优化代码”专业得多。

## 2. 机器学习基础

机器学习可以先用一句话理解：**不是人手写规则，而是让模型从样本中归纳规则**。

传统程序通常是：

```text
人工写规则 + 输入数据 -> 输出结果
```

机器学习是：

```text
输入数据 + 正确答案 -> 训练出规则/模型
训练好的模型 + 新输入 -> 预测结果
```

例如你要判断 CT 图像是否异常，传统方式可能要医生或工程师写很多规则；机器学习方式是收集大量图像和标签，让模型自己学习哪些模式和异常有关。这里最重要的不是“模型越复杂越好”，而是数据、标签、指标和泛化是否可靠。

### 2.1 监督学习

核心流程：

1. 定义任务：分类、回归、排序、检测、分割、生成。
2. 定义数据：样本、标签、特征、划分方式。
3. 定义指标：accuracy、precision、recall、F1、AUC、MAE、RMSE、mAP、Dice。
4. 训练 baseline：简单模型优先。
5. 误差分析：按类别、场景、难例、噪声标签拆解。
6. 迭代模型和数据。

常见算法：

- 线性回归：最小化均方误差。
- 逻辑回归：二分类，输出概率。
- SVM：最大间隔分类。
- 决策树：可解释，但容易过拟合。
- 随机森林：bagging 降低方差。
- GBDT / XGBoost / LightGBM：强表格数据 baseline。

这些算法是什么、怎么做：

- **线性回归是什么**：用一条直线或超平面拟合连续数值，例如预测房价、血糖值。做法是输入特征乘权重再加偏置，用 MSE 衡量预测值和真实值差距。
- **逻辑回归是什么**：虽然叫回归，但常用于二分类。它先算线性分数，再用 sigmoid 转成正类概率，适合做简单、可解释的分类 baseline。
- **SVM 是什么**：寻找一个分类边界，让两类样本离边界的间隔尽可能大。小中型数据、特征维度较高时有用，但大规模深度学习场景不常作为主模型。
- **决策树是什么**：像一系列 if-else 问题，例如“年龄 > 60 吗”“血压高吗”，一步步把样本分到叶子节点。优点是可解释，缺点是单棵树容易过拟合。
- **随机森林怎么做**：训练很多棵决策树，每棵树看不同采样数据和特征，最后投票或平均。它通过模型集成降低单棵树的方差。
- **GBDT/XGBoost/LightGBM 怎么做**：一棵树一棵树串行训练，新树重点修正前面模型的错误。表格数据上非常强，常作为深度学习以外的强 baseline。

面试高频：

- 逻辑回归为什么用 sigmoid？
- 交叉熵和 MSE 用于分类有什么区别？
- Bagging 和 Boosting 的区别？
- AUC 为什么对阈值不敏感？
- 类别不平衡怎么处理？

高频问题答案：

**逻辑回归为什么用 sigmoid？** 二分类需要输出 0 到 1 之间的概率，sigmoid 可以把任意实数映射到 `(0, 1)`。模型先算线性分数 `w^T x + b`，再经过 sigmoid 得到正类概率。

**交叉熵和 MSE 用于分类有什么区别？** MSE 衡量数值距离，分类时梯度可能不够合适；交叉熵直接惩罚正确类别概率低的情况，更符合概率建模和最大似然。因此分类通常用交叉熵。

**Bagging 和 Boosting 的区别？** Bagging 是并行训练多个模型再平均，主要降低方差，随机森林是典型例子；Boosting 是串行训练，后一个模型关注前一个模型没学好的样本，主要降低偏差，GBDT/XGBoost 是典型例子。

**AUC 为什么对阈值不敏感？** AUC 衡量模型把正样本排在负样本前面的能力，不依赖某一个固定分类阈值。它适合评估排序能力，但最终上线仍要根据业务代价选择阈值。

**类别不平衡怎么处理？** 数据侧可以重采样、补充少数类；loss 侧可以类别加权或 focal loss；评估侧不要只看 accuracy，要看 recall、precision、PR-AUC；业务侧要调阈值并看误报漏报代价。

从 0 理解：

监督学习就是“给例题和答案，让模型学会做新题”。每个样本都有输入 `x` 和标签 `y`。分类任务中，`y` 是类别；回归任务中，`y` 是连续数值；分割任务中，`y` 可能是一张 mask。

一个完整监督学习流程不是从模型开始，而是从问题开始：

```text
我要解决什么问题 -> 有没有带标签数据 -> 标签是否可信 -> 选什么指标 -> 先做简单 baseline -> 再考虑复杂模型
```

为什么 baseline 重要？因为它告诉你问题本身是否可学。如果逻辑回归或简单 CNN 已经很好，说明任务可能不需要复杂模型；如果简单模型完全学不会，要先查数据和标签，而不是马上堆更大模型。

面试时可以这样讲：

“我会先建立一个简单可靠的 baseline，再做误差分析，确认问题来自模型容量、数据质量、类别不平衡还是评估方式。这样模型迭代才有方向。”

### 2.2 训练集、验证集、测试集

关键原则：

- 测试集只能最后用。
- 验证集用于调参。
- 如果数据有时间顺序，不能随机切分造成未来信息泄漏。
- 如果同一个病人、设备、用户出现多条样本，切分时要按实体隔离，避免泄漏。

医学影像场景常见坑：

- 同一患者多张 slice 分到训练和测试，指标虚高。
- DICOM 元信息泄漏标签。
- 只在单一医院数据上测试，跨域泛化差。
- 标注者一致性差，模型上限被标签噪声限制。

从 0 理解：

训练集像教材，验证集像平时测验，测试集像期末考试。你可以用教材学习，也可以用平时测验调学习方法，但期末考试不能提前偷看。如果你反复根据测试集调参，测试集就不再能代表真实泛化能力。

数据泄漏是面试高频，因为它会让模型“看起来很强，其实作弊”。医学影像尤其容易泄漏：同一个患者的不同切片如果同时出现在训练集和测试集，模型可能学到的是这个患者或设备特征，而不是真正病灶规律。

正确思路：

```text
按患者/时间/机构隔离划分 -> 只在训练集学 -> 验证集调参 -> 最后一次测试集报告结果
```

怎么做：

1. 先确定切分单位：普通图片可按样本，医学影像通常按患者，推荐系统常按用户或时间。
2. 再确定切分比例：常见是 8:1:1 或 7:1:2，但小数据要更重视交叉验证。
3. 切分后检查标签分布，避免某一类只出现在训练集或测试集。
4. 所有归一化、特征统计、数据增强策略都只能从训练集估计，不能偷看测试集。
5. 最终测试集只用于最后报告，不用于反复调参。

高频问题答案：

**为什么不能随机切所有医学 slice？** 因为同一患者不同 slice 高度相关，随机切会让训练集和测试集出现同一患者信息，模型可能记住患者或扫描条件，导致指标虚高。

**验证集和测试集区别是什么？** 验证集用于模型选择和调参，测试集用于最终估计泛化能力。验证集可以多次看，测试集应尽量只看一次。

### 2.3 指标选择

分类：

- Accuracy：类别均衡时有用。
- Precision：预测为正的样本中有多少是真的正。
- Recall：真实正样本中有多少被找出来。
- F1：precision 和 recall 的调和平均。
- AUC：排序能力。
- PR-AUC：正样本稀少时比 ROC-AUC 更敏感。

分类指标逐项解释：

- **Accuracy 怎么算**：预测正确数 / 总样本数。类别均衡时直观，但类别极不平衡时容易骗人。
- **Precision 怎么算**：`TP / (TP + FP)`，表示模型报出的阳性里有多少是真的。适合关注误报代价的场景。
- **Recall 怎么算**：`TP / (TP + FN)`，表示真实阳性中有多少被找出来。适合关注漏报代价的场景，例如医学筛查。
- **F1 怎么算**：Precision 和 Recall 的调和平均，只有两者都高时 F1 才高。适合需要平衡误报和漏报的场景。
- **AUC 怎么理解**：看排序能力，不依赖固定阈值。模型把阳性排在阴性前面的概率越高，AUC 越高。
- **PR-AUC 怎么理解**：在正样本很少时，PR 曲线比 ROC 更能反映模型是否真的找到了少数正例。

分割：

- IoU：预测区域和真实区域交并比。
- Dice：医学分割常用，和 F1 类似。
- Hausdorff distance：边界误差。

分割指标逐项解释：

- **IoU 怎么算**：预测 mask 和真实 mask 的交集面积 / 并集面积。越接近 1，重叠越好。
- **Dice 怎么算**：`2 * 交集 / (预测面积 + 真实面积)`。它对小目标更敏感，医学分割中常用来评估器官或病灶区域。
- **Hausdorff distance 是什么**：衡量两个边界最坏情况下相距多远。Dice 高不代表边界完全好，所以边界敏感任务会补充这个指标。

生成：

- BLEU、ROUGE、BERTScore：只适合辅助。
- Human eval：常常更重要。
- 事实性、引用准确率、安全性、任务完成率。

生成指标逐项解释：

- **BLEU/ROUGE 是什么**：基于 n-gram 重叠的文本相似度指标，机器翻译和摘要常见，但不能可靠判断事实是否正确。
- **BERTScore 是什么**：用语义 embedding 比较生成文本和参考答案，比纯字面重叠更灵活，但仍然只是辅助。
- **Human eval 怎么做**：让人工按标准打分，例如事实正确、表达清楚、是否有害、是否完成任务。大模型应用中经常不可替代。
- **引用准确率是什么**：回答中引用的证据是否真的支持结论。RAG 场景必须看它。

上线指标：

- 延迟：P50、P95、P99。
- 吞吐：QPS、tokens/s。
- 成本：每千 token 成本、GPU 利用率、显存占用。
- 稳定性：错误率、超时率、重试率。
- 业务指标：转化率、医生节省时间、报告一致性。

怎么选指标：

1. 先看任务类型：分类、回归、分割、生成、检索、推理服务。
2. 再看业务代价：漏报更严重还是误报更严重。
3. 再看数据分布：类别是否均衡，正样本是否稀少。
4. 最后看上线约束：延迟、吞吐、成本、稳定性是否达标。

高频问题答案：

**Precision 和 Recall 怎么取舍？** Precision 高表示报出来的正例更可信，Recall 高表示少漏掉真实正例。医疗筛查通常更重 recall，避免漏诊；自动处置类系统通常也要重 precision，避免误操作。

**Dice 和 IoU 区别是什么？** 二者都衡量预测区域和真实区域重叠。Dice 对小目标更友好，医学分割常用；IoU 更严格，目标检测和分割也常见。

**为什么只看平均值不够？** 平均值会掩盖长尾问题。线上系统要看分桶指标和 P95/P99，例如少数复杂病例或长 prompt 请求可能表现很差。

从 0 理解：

指标是“你怎么判断模型好不好”。不同任务不能乱用同一个指标。比如癌症筛查里，漏掉真正阳性可能代价很高，所以 recall 很重要；垃圾邮件过滤里，把正常邮件误判成垃圾也很糟，所以 precision 也重要。

Accuracy 在类别不平衡时可能骗人。假设 1000 个样本里只有 10 个阳性，模型全部预测阴性，也有 99% accuracy，但它没有发现任何真正阳性。这时要看 recall、precision、F1 或 PR-AUC。

面试表达：

“我不会只看一个总体指标，而会结合业务代价选择指标。医疗场景通常更关注漏诊风险，所以会重点看 recall、敏感性、特异性，并结合医生审核流程控制误报。”

### 2.4 特征工程和数据工程

表格数据：

- 缺失值处理。
- 类别编码。
- 数值归一化。
- 时间窗口特征。
- 交叉特征。

表格数据怎么做：

- **缺失值处理**：可以删除、填均值/中位数、填特殊值，或增加“是否缺失”特征。医疗数据中缺失本身可能有含义，不能随便填。
- **类别编码**：把性别、科室、诊断类型等离散值转成数字。低基数可 one-hot，高基数可 target encoding 或 embedding。
- **数值归一化**：把不同量纲的数值缩放到相近范围，帮助梯度优化。例如年龄、血压、费用尺度差异很大。
- **时间窗口特征**：统计过去 7 天、30 天、半年内的次数、均值、最大值等，常用于风控、医疗随访、用户行为。
- **交叉特征**：组合两个或多个特征，例如“年龄段 × 疾病类型”。树模型常自动学，线性模型更依赖手工交叉。

图像数据：

- resize、crop、normalize。
- 数据增强：flip、rotate、color jitter、mixup、cutmix。
- 医学影像：窗宽窗位、spacing、重采样、HU 值截断。

图像数据怎么做：

- **resize/crop 是什么**：把图像变成模型需要的固定尺寸。resize 可能改变比例，crop 可能裁掉关键区域，所以医学影像要谨慎。
- **normalize 是什么**：减均值、除标准差，让输入分布稳定，帮助训练收敛。训练和推理必须一致。
- **flip/rotate/color jitter 是什么**：通过翻转、旋转、颜色扰动制造更多样本，提高泛化。但医疗图像不能做会改变临床含义的增强。
- **mixup/cutmix 是什么**：把两张图或区域混合，标签也混合，用于增强泛化。医学诊断场景要确认混合后的样本是否仍有合理语义。
- **HU 值截断是什么**：CT 像素有物理意义，范围很大。截断到关注窗口可以突出目标组织，例如脑窗、骨窗。
- **spacing/重采样是什么**：不同 CT 扫描体素间距不同。重采样让空间尺度一致，3D 模型尤其需要。

文本数据：

- tokenization。
- 清洗 HTML、特殊符号、PII。
- 长文本分块。
- instruction / response 格式统一。

文本数据怎么做：

- **tokenization 是什么**：把文本切成模型认识的 token id。不同 tokenizer 对中文、英文、代码的切分差异会影响长度和成本。
- **清洗 HTML/特殊符号**：去掉无关标签、乱码和重复模板，避免模型学习噪声。
- **PII 清洗是什么**：删除或脱敏姓名、身份证、手机号、病历号等敏感信息，避免隐私泄漏。
- **长文本分块怎么做**：按标题、段落或语义切 chunk，配合 overlap，供 RAG 检索或长文处理。
- **instruction/response 格式统一**：微调数据必须统一模板，否则模型会学到混乱格式，影响输出稳定性。

面试重点：

- 数据质量通常比模型结构更影响上限。
- 任何增强都要符合业务语义，比如医学图像不能随意做会改变左右含义的增强。

怎么做：

1. 先定义原始数据 schema：字段、类型、单位、缺失规则。
2. 清洗异常值和重复样本，记录被删除或修正的规则。
3. 按任务做标准化：图像 normalize，文本 tokenize，表格编码类别和数值。
4. 训练阶段做数据增强，但验证和测试阶段只做确定性预处理。
5. 保存预处理配置，保证训练、验证、线上推理一致。

高频问题答案：

**为什么训练和线上预处理必须一致？** 如果训练时输入归一化到某个分布，线上却用另一套处理，模型看到的数据分布就变了，结果会退化。

**医学影像为什么要窗宽窗位？** CT 的 HU 值范围很大，不同组织关注范围不同。窗宽窗位把关注范围映射到可学习的灰度区间，有助于突出脑组织、骨窗或出血等结构。

从 0 理解：

模型吃进去的不是“现实世界”，而是被处理成数字的数据。特征工程就是把原始数据变成模型更容易学习的形式。深度学习减少了手工特征，但没有消灭数据工程。图片要归一化，文本要 tokenization，医学影像要处理窗宽窗位，表格要处理缺失值和类别变量。

数据工程的核心逻辑：

```text
原始数据 -> 清洗 -> 标准化 -> 划分 -> 加载 -> 增强 -> 送入模型
```

如果模型效果不好，很多时候不是网络结构不够先进，而是数据质量有问题：标签错、类别偏、预处理不一致、训练和线上输入格式不同。面试中能主动从数据侧排查，会显得更接近真实工程。

## 3. 深度学习基础

深度学习是机器学习的一类方法，它的特点是用多层神经网络自动学习特征。传统机器学习常常需要人设计特征，例如“边缘数量、纹理强度、某个统计值”；深度学习则让模型从像素、文本 token、音频波形中逐层学出特征。

可以把神经网络理解成很多层函数的组合：

```text
输入 x -> 第 1 层变换 -> 激活函数 -> 第 2 层变换 -> ... -> 输出预测
```

训练时，模型先给出预测，loss 衡量预测和真实答案的差距，反向传播计算每个参数应该怎么调，优化器更新参数。这个循环重复很多次后，模型就从“随机猜”变成“学到规律”。

深度学习的主线逻辑：

```text
表示能力：网络能不能表达复杂规律
优化能力：网络能不能被训练好
泛化能力：网络能不能在没见过的数据上表现好
工程能力：网络能不能在资源限制下稳定运行
```

### 3.1 神经网络基本结构

基本模块：

- Linear：矩阵乘加。
- Activation：ReLU、GELU、SiLU。
- Normalization：BatchNorm、LayerNorm、RMSNorm。
- Dropout：训练时随机置零，减少过拟合。
- Residual：缓解深层网络训练困难。

基本模块逐项解释：

- **Linear 怎么做**：输入 `x` 乘权重 `W` 再加偏置 `b`，即 `y = xW + b`。它负责把特征从一个维度映射到另一个维度。
- **Activation 怎么做**：在线性变换后加非线性函数。没有激活函数，多层线性网络仍然等价于一层线性网络，表达能力不足。
- **ReLU/GELU/SiLU 区别**：ReLU 简单快；GELU 更平滑，Transformer 常用；SiLU 在一些现代网络中效果好。选择通常跟架构经验和实验结果有关。
- **Normalization 怎么做**：对中间激活做归一化，让数值分布更稳定，训练更容易。CNN 常见 BatchNorm，Transformer 常见 LayerNorm/RMSNorm。
- **Dropout 怎么做**：训练时随机丢弃部分激活，防止模型过度依赖某些特征。推理时关闭。
- **Residual 怎么做**：把输入直接加到输出上，例如 `y = F(x) + x`。它帮助深层网络传播信息和梯度。

常见问题：

- ReLU 为什么比 sigmoid 更适合深层网络？
- BatchNorm 和 LayerNorm 区别？
- Transformer 为什么主要用 LayerNorm？
- 残差连接为什么有用？

高频问题答案：

**ReLU 为什么比 sigmoid 更适合深层网络？** sigmoid 在输入很大或很小时梯度接近 0，深层网络容易梯度消失；ReLU 在正半轴梯度恒为 1，传播更直接，计算也更简单。但 ReLU 也可能出现 dead ReLU，所以后来有 LeakyReLU、GELU、SiLU 等变体。

**BatchNorm 和 LayerNorm 区别？** BatchNorm 在 batch 维度上统计均值方差，常用于 CNN，但小 batch 时统计不稳定；LayerNorm 在单个样本的特征维度上归一化，不依赖 batch size，适合 Transformer 和变长序列。

**Transformer 为什么主要用 LayerNorm？** NLP 和大模型训练中 batch、序列长度、并行切分都比较复杂，BatchNorm 的 batch 统计不稳定且不方便自回归推理；LayerNorm 对每个 token 的 hidden dimension 做归一化，更稳定也更容易部署。

**残差连接为什么有用？** 残差让网络学习 `F(x)+x`，如果新层暂时学不到有用变换，也能近似恒等映射。这样信息和梯度更容易穿过深层网络，缓解退化问题。

从 0 理解：

Linear 层负责做线性变换，但只有线性层叠很多层，本质上仍然等价于一个线性变换，表达能力有限。激活函数引入非线性，让网络能拟合复杂模式。Normalization 让中间激活值分布更稳定，训练更容易。Residual 让深层网络可以学习“在原输入基础上做一点修改”，缓解深层网络退化。

一个常见网络块可以理解为：

```text
输入 -> 线性/卷积 -> 归一化 -> 激活 -> 残差连接 -> 输出
```

面试时不要只背模块名，要说清模块的作用：Linear/Conv 负责变换，Activation 负责非线性，Normalization 负责稳定训练，Dropout/正则化负责降低过拟合，Residual 负责改善深层优化。

### 3.2 反向传播

要能讲清：

- 前向计算得到 loss。
- 反向传播根据链式法则计算每个参数的梯度。
- 优化器根据梯度更新参数。
- autograd 会记录计算图，动态框架如 PyTorch 每次 forward 构建图。

面试回答模板：

“反向传播不是一个独立的魔法算法，而是链式法则在计算图上的高效实现。每个算子定义 forward 和 backward，框架从 loss 节点反向遍历图，把上游梯度乘以局部梯度，累积到叶子参数。”

怎么做：

1. forward：输入经过模型得到预测。
2. loss：把预测和真实标签比较，得到一个标量损失。
3. backward：从 loss 反向遍历计算图，计算每个参数梯度。
4. step：优化器根据梯度更新参数。
5. zero grad：清空梯度，避免和下一轮累积混在一起。

高频问题答案：

**为什么 loss 必须通常是标量？** 反向传播需要从一个目标开始计算梯度，训练时通常把 batch 上的 loss 做 mean 或 sum，得到一个标量优化目标。

**为什么要清空梯度？** PyTorch 默认梯度累积，方便 gradient accumulation。如果每个 batch 前不清空，梯度会把历史 batch 混进来，更新方向错误。

**detach 有什么用？** `detach()` 会切断计算图，让后续操作不再把梯度传回原来的 tensor，常用于停止梯度、缓存中间结果或日志记录。

从 0 理解：

你可以把模型看成一条流水线，每个节点都做一个小计算。前向传播从输入一路算到输出；反向传播从 loss 往回走，问每个参数：“如果你稍微变大一点，loss 会变大还是变小？”这个变化方向就是梯度。

PyTorch 的 autograd 会在 forward 时记录计算图。只要 tensor 参与了需要梯度的计算，框架就知道 backward 时该怎么把梯度传回去。面试中如果问 `detach()`，它的含义就是把某个 tensor 从计算图中断开，后续梯度不再传回它之前的计算。

训练循环的本质：

```text
forward 预测 -> 计算 loss -> backward 求梯度 -> optimizer 更新参数 -> 清空梯度 -> 下一批数据
```

### 3.3 CNN

CNN 是什么：

CNN，全称 Convolutional Neural Network，卷积神经网络。它主要用于图像、医学影像、视频帧、局部时序信号等具有“局部结构”的数据。普通全连接网络会把每个像素都和所有参数连接起来，参数量很大，也不利用图像的空间结构；CNN 用一个小窗口在图像上滑动，只看局部区域，因此更适合提取边缘、纹理、形状和局部病灶。

CNN 的基本计算逻辑：

```text
输入图像/特征图 -> 卷积提取局部特征 -> 激活函数引入非线性 -> 下采样扩大感受野 -> 更深层组合高级语义 -> 分类/检测/分割输出
```

核心概念逐个解释：

- **卷积核是什么**：卷积核是一个很小的可学习矩阵，例如 `3x3` 或 `5x5`。它在图像上滑动，每到一个位置，就和局部像素做乘加，得到一个输出值。训练前卷积核是随机的，训练后可能学到边缘、角点、纹理、器官边界等模式。
- **卷积怎么做**：假设输入是 `H x W` 图像，一个 `3x3` 卷积核每次覆盖 9 个像素，做 9 次乘法再求和。多个卷积核会产生多个输出通道，也就是多张 feature map。
- **stride 是什么**：stride 是卷积核每次滑动的步长。`stride=1` 表示每次移动 1 个像素，输出更细；`stride=2` 表示每次移动 2 个像素，输出尺寸变小，计算量下降，也起到下采样作用。
- **padding 是什么**：padding 是在图像边缘补 0 或其他值。没有 padding 时，卷积后尺寸会变小，边缘信息被快速丢掉；有 padding 时，可以保持输出尺寸，常见 `3x3` 卷积配 `padding=1`。
- **dilation 是什么**：dilation 是空洞卷积，在卷积核元素之间插入间隔。它不明显增加参数量，却能看更大范围，适合分割、语义理解等需要更大上下文的任务。
- **感受野是什么**：某个输出位置能“看到”的原始输入范围叫感受野。网络越深、下采样越多、卷积核越大，感受野通常越大。医学影像中，太小的感受野可能只能看到局部纹理，看不到病灶和周围结构关系。
- **参数共享是什么**：同一个卷积核在整张图上反复使用，所以无论目标出现在左上角还是右下角，模型都可以用同一组参数识别类似模式。这让 CNN 参数更少，也更适合图像。
- **平移等变性是什么**：如果输入图像中的目标平移，卷积特征也会相应平移。注意这不是完全“平移不变”，而是特征位置跟着移动。Pooling 或全局平均池化可以进一步增强平移不变性。
- **Pooling 是什么**：Pooling 是下采样操作，常见 max pooling 和 average pooling。max pooling 取局部最大值，保留最强响应；average pooling 取平均值，保留整体趋势。它可以降低尺寸、减少计算、扩大后续层感受野。
- **典型结构是什么**：LeNet 是早期 CNN；AlexNet 证明深度 CNN 在大规模图像分类上的威力；VGG 使用大量 `3x3` 卷积堆叠；ResNet 引入残差连接解决深层网络训练难；U-Net 用编码器-解码器和 skip connection 做精细分割。

典型结构进一步解释：

- **LeNet 怎么工作**：LeNet 用卷积、池化、全连接完成手写数字识别。它展示了 CNN 的基本套路：先用卷积提局部特征，再下采样，最后分类。面试中提 LeNet，重点是说明 CNN 的早期基本形态。
- **AlexNet 为什么重要**：AlexNet 更深、更宽，使用 ReLU、Dropout 和 GPU 训练，在 ImageNet 上取得突破。它证明了大数据 + GPU + 深度 CNN 的组合有效。
- **VGG 怎么工作**：VGG 用很多小的 `3x3` 卷积堆叠代替大卷积核。两个 `3x3` 堆叠能扩大感受野，同时增加非线性，结构简单规整，适合作为理解 CNN backbone 的模板。
- **ResNet 怎么工作**：ResNet 在若干卷积层外加 shortcut，把输入直接加到输出。网络学习的是残差变化，不必每层都重新学习完整映射，因此可以训练很深。
- **U-Net 怎么工作**：U-Net 左边编码器逐步下采样提语义，右边解码器逐步上采样恢复分辨率，中间用 skip connection 传递浅层细节。它不是只判断整张图类别，而是输出每个像素/体素的类别，所以适合分割。

CNN 模块实际怎么搭：

1. 输入图像先统一尺寸和归一化。
2. 用多层卷积 + 激活提取低级到高级特征。
3. 通过 stride 或 pooling 下采样，扩大感受野并降低计算量。
4. 分类任务接 global average pooling 或全连接层；分割任务接上采样解码器。
5. 训练时选择合适 loss：分类用交叉熵，分割常用 Dice/BCE + Dice。
6. 评估时按任务看 accuracy/AUC 或 Dice/IoU，而不是只看 loss。

医学影像项目常用：

- **2D U-Net 怎么做**：把 CT 或 MRI 的每张 slice 当作 2D 图像输入，输出对应 slice 的分割 mask。优点是简单、省显存、训练快；缺点是切片间上下文利用不足。
- **3D U-Net 怎么做**：把连续多张 slice 组成一个 3D volume，使用 3D 卷积直接学习体数据。优点是能利用上下文；缺点是显存消耗大、训练慢、对数据和标注要求更高。
- **CNN-LSTM 怎么做**：先用 CNN 提取每张 slice 的特征，再用 LSTM 建模 slice 序列关系。它介于 2D 和 3D 之间，适合想利用切片顺序但显存不足以训练完整 3D 模型的情况。
- **MONAI / nnU-Net 是什么**：它们是医学影像深度学习工具链。MONAI 提供医学影像预处理、增强、模型、loss、metrics；nnU-Net 则强调自动配置预处理、网络结构和训练策略，是医学分割的强 baseline。

高频问题：

- 1x1 卷积有什么用？
- ResNet 为什么能训练很深？
- U-Net 为什么适合医学分割？
- 2D 和 3D 医学图像模型取舍？

高频问题答案：

**1x1 卷积有什么用？**

1x1 卷积不看邻域空间，只在通道维度上做线性组合。它常用于通道升维、降维和特征融合。例如输入有 256 个通道，用 1x1 卷积变成 64 个通道，可以减少后续 3x3 卷积计算量；也可以把不同通道的信息重新组合。面试中可以说：1x1 卷积是“逐像素的通道混合器”。

**ResNet 为什么能训练很深？**

深层网络直接学习 `H(x)` 很难，ResNet 改为学习残差 `F(x)=H(x)-x`，输出变成 `F(x)+x`。如果某些层暂时学不到有用变化，至少可以接近恒等映射，让信息和梯度更容易穿过深层网络。它缓解了深层网络退化和梯度传播困难，因此可以训练几十层甚至上百层。

**U-Net 为什么适合医学分割？**

医学分割既要知道“这里是不是病灶”，也要知道“边界在哪里”。U-Net 的编码器负责下采样，提取高级语义和大范围上下文；解码器负责上采样，恢复空间分辨率；skip connection 把浅层细节直接传到解码器，帮助恢复边界。因此它特别适合器官、病灶、伪影掩码这类像素级任务。

**2D 和 3D 医学图像模型怎么取舍？**

如果数据量少、显存有限、标注是逐 slice 的，可以先用 2D U-Net 做 baseline。如果病灶跨切片明显、上下文很重要、显存和数据允许，可以用 3D U-Net。如果想折中，可以用 2.5D，把相邻几张 slice 当作多通道输入，或者用 CNN-LSTM 建模序列。面试中不要说“3D 一定更好”，要结合数据、显存、标注和指标。

从 0 理解：

CNN 适合图像，因为图像有局部结构：一个像素和周围像素关系更强。卷积核像一个小窗口，在图像上滑动，提取边缘、纹理、形状等局部特征。浅层卷积常学到边缘和颜色，深层卷积会组合成更抽象的结构。

U-Net 的直觉是“先压缩理解全局，再恢复精细位置”。编码器逐步下采样，获得更大感受野；解码器逐步上采样，恢复空间分辨率；skip connection 把浅层细节传给解码器，所以适合医学分割这类既要语义又要边界的任务。

2D 和 3D 的取舍：

- 2D 模型省显存、数据量需求较低、训练简单，但可能丢失切片间上下文。
- 3D 模型能利用体数据结构，更适合 CT/MRI，但显存压力大，标注和预处理更复杂。

面试可说：“我会先根据数据规模、标注形式、显存和目标指标选择 2D/2.5D/3D，而不是默认 3D 一定更好。”

### 3.4 RNN / LSTM / GRU

现在不如 Transformer 常用，但面试可能会问：

- RNN 处理序列，但长距离依赖困难。
- LSTM 通过门控缓解梯度消失。
- GRU 是更轻量的门控结构。

对比 Transformer：

- RNN 串行处理，难并行。
- Transformer self-attention 可以并行处理序列，但注意力复杂度是 O(n^2)。

高频问题答案：

**RNN 为什么难处理长距离依赖？** RNN 把历史信息压缩进 hidden state，序列越长，早期信息越容易被覆盖；反向传播跨很多时间步，梯度也容易消失或爆炸。

**LSTM 的门控在做什么？** LSTM 用输入门、遗忘门、输出门控制信息写入、保留和输出。它不是简单地每一步覆盖 hidden state，而是有一个 cell state 帮助长期信息流动。

**为什么 Transformer 取代 RNN 成为主流？** Transformer 的 self-attention 允许任意 token 直接交互，训练时可以并行处理整段序列，更适合 GPU。代价是 attention 对序列长度通常是平方复杂度。

从 0 理解：

RNN 的想法是“按顺序读”，每读一个 token，就把之前的信息压缩进 hidden state。问题是序列很长时，早期信息很难完整保留下来，而且计算必须一步接一步，GPU 并行效率不高。

LSTM/GRU 用门控机制决定哪些信息保留、哪些遗忘，缓解长距离依赖。但大模型时代主流转向 Transformer，因为 self-attention 可以让任意两个 token 直接建立联系，也更适合并行计算。

### 3.5 Transformer

核心公式：

```text
Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) V
```

关键组件：

- Token embedding。
- Positional encoding / RoPE。
- Multi-head attention。
- Feed-forward network。
- LayerNorm / RMSNorm。
- Residual connection。

为什么多头：

- 不同 head 可以关注不同子空间和关系。
- 工程上是把 hidden dimension 切成多个 head 并行计算。

为什么除以 sqrt(d_k)：

- QK 点积维度越大，方差越大，softmax 容易饱和，梯度变小。
- 缩放可以稳定训练。

复杂度：

- Attention 计算量和显存通常随序列长度平方增长。
- 长上下文优化关注稀疏注意力、滑窗、分块、KV cache、paged attention、上下文压缩。

怎么做：

1. 输入 token 先变成 embedding。
2. 通过线性层生成 Q、K、V。
3. 计算 `QK^T / sqrt(d_k)` 得到注意力分数。
4. 加 mask，防止看 padding 或未来 token。
5. softmax 得到权重。
6. 权重乘 V 得到上下文表示。
7. 经过 FFN、残差和归一化输出。

高频问题答案：

**Q、K、V 分别是什么？** Q 是 query，表示当前 token 想找什么；K 是 key，表示每个 token 能被匹配的特征；V 是 value，表示被关注后提供的信息。Q 和 K 算相似度，得到权重，再对 V 加权求和。

**为什么要除以 `sqrt(d_k)`？** 维度越大，点积分数方差越大，softmax 容易变得过于尖锐，梯度不稳定。除以 `sqrt(d_k)` 可以稳定数值范围。

**Multi-head attention 为什么有用？** 多个 head 可以在不同子空间关注不同关系，例如局部依赖、长距离指代、句法关系。工程上是把 hidden dimension 拆成多个 head 并行计算。

**Attention 复杂度为什么是 O(n^2)？** 每个 token 都要和其他 token 计算相似度，长度为 n 就有 n×n 个关系，所以长上下文会带来显存和计算压力。

从 0 理解：

Transformer 的核心是 self-attention。假设一句话里有很多 token，每个 token 都想知道“我应该关注其他哪些 token”。Q 可以理解成“我在找什么”，K 是“我有什么特征可被匹配”，V 是“如果你关注我，我能提供什么信息”。Q 和 K 算相似度，softmax 变成权重，再对 V 加权求和。

一个 Transformer block 通常是：

```text
输入 token 表示
-> LayerNorm
-> Multi-Head Self-Attention
-> Residual
-> LayerNorm
-> Feed Forward Network
-> Residual
-> 输出 token 表示
```

为什么大模型推理需要 KV cache？生成第 100 个 token 时，前 99 个 token 的 K/V 已经算过，如果每一步都重算历史，会浪费大量计算。KV cache 把历史 K/V 存起来，新 token 只和缓存交互，从而加速自回归生成。

面试表达：

“Transformer 的优势是通过 attention 建模长距离依赖并适合并行训练；代价是 attention 对序列长度通常是平方复杂度。训练和推理优化会围绕矩阵乘、attention、KV cache 和显存管理展开。”

### 3.6 损失函数

分类：

- Cross Entropy。
- Focal Loss：类别极不平衡时降低易分类样本权重。

分类 loss 怎么做：

- **Cross Entropy 怎么做**：模型输出 logits，softmax 变成概率，然后惩罚正确类别概率太低的情况。PyTorch 中常用 `nn.CrossEntropyLoss`，它内部会处理 log-softmax，不需要你先手动 softmax。
- **Focal Loss 怎么做**：在交叉熵基础上给易分类样本更低权重，让模型关注难样本。目标检测、医学异常检测、类别极不平衡任务常用。

回归：

- MSE：对大误差敏感。
- MAE：更鲁棒。
- Huber：折中。

回归 loss 怎么做：

- **MSE 怎么做**：预测值和真实值差的平方再平均。大误差会被平方放大，所以对离群点敏感。
- **MAE 怎么做**：预测值和真实值差的绝对值再平均。对离群点更鲁棒，但在 0 附近梯度不如 MSE 平滑。
- **Huber 怎么做**：小误差用 MSE，大误差用 MAE，兼顾平滑和鲁棒性。

分割：

- Dice Loss。
- BCE + Dice。
- Tversky / Focal Tversky。

分割 loss 怎么做：

- **Dice Loss 怎么做**：用 `1 - Dice` 作为损失，直接优化预测 mask 和真实 mask 的重叠。小病灶、前景稀少时比单纯 BCE 更合适。
- **BCE + Dice 怎么做**：BCE 负责像素级分类，Dice 负责整体区域重叠。医学分割中常把二者加权相加。
- **Tversky / Focal Tversky 怎么做**：给 FP 和 FN 不同权重，适合误报和漏报代价不同的场景。Focal 版本进一步关注难分样本。

语言模型：

- next token prediction 的交叉熵。
- perplexity 是交叉熵的指数形式，越低通常代表语言建模越好。

语言模型 loss 怎么做：

- **next token prediction 怎么做**：输入一段 token，模型在每个位置预测下一个 token。训练标签就是输入序列右移一位。
- **Perplexity 怎么算**：通常是 `exp(cross_entropy)`。它越低，说明模型对真实下一个 token 越不困惑，但它不能直接代表事实性或安全性。

高频问题答案：

**交叉熵怎么理解？** 它惩罚模型给正确答案的概率太低。正确类别概率越接近 1，loss 越小；越接近 0，loss 越大。

**Focal Loss 解决什么问题？** 在类别极不平衡时，大量容易分类的负样本会主导训练。Focal Loss 降低易分类样本权重，让模型更关注难样本和少数类。

**Dice Loss 为什么适合医学分割？** 医学病灶常常很小，背景像素很多。Dice 直接优化预测区域和真实区域的重叠程度，能减轻背景像素过多带来的影响。

**Perplexity 是什么？** 困惑度可以理解成模型对下一个 token 的平均不确定性。越低表示模型越能预测文本，但低困惑度不一定代表问答事实性好。

从 0 理解：

loss 是模型训练的“方向盘”。模型预测错了，loss 变大；预测对了，loss 变小。优化器不会直接知道“医学图像哪里异常”，它只知道怎么让 loss 下降。因此 loss 设计必须和任务目标一致。

分类用交叉熵，是因为我们希望正确类别概率高；分割常用 Dice Loss，是因为医学分割关注区域重叠，尤其在病灶很小、前景很少时，单纯 BCE 可能被大量背景像素主导；语言模型用 next token 交叉熵，因为训练目标是预测下一个 token。

面试中如果问“为什么换 loss”，回答要围绕数据分布和业务代价：类别不平衡、前景稀疏、异常样本重要、边界质量重要等。

### 3.7 正则化和泛化

常用方法：

- 数据增强。
- weight decay。
- dropout。
- label smoothing。
- early stopping。
- mixup / cutmix。
- 模型集成。

正则化方法逐项解释：

- **数据增强怎么做**：对训练样本做合理变换，例如图像旋转、裁剪、噪声、文本改写。它让模型看到更多变化，减少死记训练集。
- **weight decay 怎么做**：更新参数时额外惩罚过大的权重，让模型更平滑，降低过拟合。
- **dropout 怎么做**：训练时随机置零部分激活，推理时关闭。它像训练多个子网络的近似集成。
- **label smoothing 怎么做**：不把正确标签设为 1，而是设为例如 0.9，其余类别分一点概率，避免模型过度自信。
- **early stopping 怎么做**：监控验证集，如果连续若干轮不提升就停止，并保存最佳模型。
- **mixup/cutmix 怎么做**：混合两张图或两个样本及其标签，迫使模型学习更平滑的决策边界。
- **模型集成怎么做**：多个模型投票或平均输出，通常提升稳定性，但推理成本更高。

不要只会列方法，要能解释什么时候用：

- 小数据集：增强、迁移学习、冻结 backbone、强验证。
- 类别不平衡：重采样、加权 loss、focal loss、调阈值。
- 分布变化：domain adaptation、持续评估、线上反馈。

高频问题答案：

**Dropout 怎么工作？** 训练时随机把一部分神经元输出置零，迫使网络不要过度依赖某些特征；推理时不再随机置零，而是使用完整网络。

**Weight decay 和 L2 正则是什么关系？** 二者都倾向于限制权重过大。AdamW 中 weight decay 从梯度更新中解耦，比在 Adam 里直接加 L2 更符合预期。

**数据增强为什么能提升泛化？** 它让模型看到同一语义的不同变化，减少对训练集偶然细节的依赖。但增强必须符合任务语义，医疗图像不能随意做会改变诊断含义的变换。

**Early stopping 怎么做？** 监控验证集指标，如果连续若干轮不再提升，就停止训练并回滚到最佳 checkpoint，避免继续过拟合。

从 0 理解：

泛化就是模型在没见过的数据上也能表现好。过拟合是模型把训练集记得太死，包括噪声也记住了。正则化的目标是让模型别太依赖训练集中的偶然细节。

常见判断：

```text
训练集差，验证集也差 -> 欠拟合，模型或训练不够
训练集好，验证集差 -> 过拟合，泛化不够
训练集好，验证集好，线上差 -> 数据分布或工程链路不一致
```

可面试的关键不是背方法，而是能根据现象选方法。例如小数据集优先数据增强和迁移学习；类别极不平衡考虑 focal loss、重采样和阈值调整；跨医院医学影像效果差要考虑 domain shift 和外部验证。

## 4. PyTorch 实战能力

PyTorch 是把深度学习理论落成代码的主要工具之一。0 基础学习 PyTorch，不要先沉迷 API 列表，而要抓住四件事：

```text
Tensor 表示数据
Module 表示模型
Autograd 负责求梯度
Optimizer 负责更新参数
```

一个 PyTorch 项目通常由 Dataset/DataLoader、model、loss、optimizer、train/eval loop、checkpoint、日志和配置组成。能独立写出并 debug 这个闭环，是从“看过深度学习”到“能面试”的分界线。

### 4.1 Tensor 基础

必须熟悉：

- shape、stride、contiguous。
- view 和 reshape。
- permute / transpose。
- broadcasting。
- dtype：fp32、fp16、bf16、int8。
- device：CPU / CUDA。

Tensor 相关概念逐项解释：

- **shape 怎么看**：每次写模型先打印关键 tensor shape，确认 batch、通道、序列、hidden 维是否符合预期。
- **stride 怎么看**：stride 决定逻辑索引如何映射到底层内存。`permute` 后 stride 会变，可能不 contiguous。
- **contiguous 是什么**：内存按逻辑顺序连续排列。很多底层算子或 `view` 要求 contiguous。
- **view 和 reshape 区别**：`view` 要求连续内存；`reshape` 会尽量返回 view，不行时可能复制数据。
- **permute/transpose 怎么做**：交换维度顺序。例如图像从 `[H,W,C]` 转成 PyTorch 常用 `[C,H,W]`。
- **broadcasting 怎么做**：把小 tensor 虚拟扩展到大 shape 参与运算，例如 bias `[H]` 加到 `[B,H]`。
- **dtype 怎么选**：fp32 稳定，fp16/bf16 省显存提速，int8/int4 多用于量化推理。
- **device 怎么管**：输入、模型、loss 相关 tensor 必须在同一设备上，否则会报 device mismatch。

面试高频：

- `view` 为什么要求 tensor contiguous？
- `permute` 后为什么可能需要 `contiguous()`？
- broadcasting 会不会真的复制数据？
- fp16 和 bf16 区别？

高频问题答案：

**shape 是什么？** shape 表示 tensor 每个维度的大小，例如 `[B, C, H, W]` 是 batch、通道、高度、宽度。多数 PyTorch bug 都可以先从 shape 对不对查起。

**stride 是什么？** stride 表示沿某个维度移动一步，在底层存储中要跳过多少元素。`permute` 可能只改变 stride，不移动真实数据。

**为什么 `view` 要求 contiguous？** `view` 只是按新的 shape 解释同一块连续内存。如果内存布局不连续，无法简单重解释，所以要先 `contiguous()` 或用更灵活的 `reshape()`。

**broadcasting 会复制数据吗？** 通常不会真实复制，而是通过 stride 或虚拟扩展让计算看起来形状一致。但某些后续操作可能触发实际 materialize，需要关注显存。

**fp16 和 bf16 区别？** fp16 精度位更多但指数范围小，容易溢出/下溢；bf16 指数范围接近 fp32，更稳定，常用于大模型训练。

从 0 理解：

Tensor 就是多维数组，是 PyTorch 中数据和中间结果的基本形式。shape 表示它有几维、每维多大；dtype 表示数字精度；device 表示放在 CPU 还是 GPU。

`stride` 可以理解成“沿某一维走一步，在内存里要跳多远”。`permute` 可能只改变视图，不真正重排内存，所以 tensor 可能变成 non-contiguous；`view` 要求内存连续，因此常常需要 `contiguous()`。这类问题在面试中很常见，因为它连接了 PyTorch API 和底层内存布局。

混合精度中，fp16 更省显存、吞吐高，但数值范围小；bf16 数值范围接近 fp32，更稳定，但硬件支持要看设备。大模型训练常偏好 bf16。

### 4.2 训练循环

标准流程：

```python
for batch in loader:
    optimizer.zero_grad(set_to_none=True)
    with torch.autocast(device_type="cuda", dtype=torch.float16):
        logits = model(batch["x"])
        loss = criterion(logits, batch["y"])
    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()
```

要点：

- `zero_grad(set_to_none=True)` 可以减少无意义写入。
- AMP 混合精度提升吞吐并降低显存。
- 梯度裁剪用于缓解梯度爆炸。
- gradient accumulation 用于模拟更大 batch。

每行代码在做什么：

- `optimizer.zero_grad(set_to_none=True)`：清空上一轮梯度。
- `autocast`：让部分计算自动使用低精度，节省显存和提升速度。
- `logits = model(...)`：前向传播得到预测。
- `loss = criterion(...)`：计算预测和标签之间的差距。
- `scaler.scale(loss).backward()`：缩放 loss 后反向传播，避免 fp16 梯度下溢。
- `scaler.step(optimizer)`：如果梯度正常，就执行参数更新。
- `scaler.update()`：动态调整缩放因子。

高频问题答案：

**为什么训练时要 `model.train()`，评估时要 `model.eval()`？** Dropout 和 BatchNorm 在训练和推理行为不同。`train()` 开启训练行为，`eval()` 固定推理行为。

**为什么评估要用 `torch.no_grad()`？** 评估不需要计算梯度，关闭 autograd 可以减少显存和加速。

**梯度累积怎么做？** 多个 mini-batch 的 loss 先 backward 累积梯度，每隔 `accum_steps` 再 optimizer step，相当于模拟更大 batch。

从 0 理解：

训练循环就是反复让模型做题、对答案、改参数。每一批 batch 都经历同样流程：

```text
取数据 -> 前向预测 -> 算 loss -> 反向传播 -> 参数更新 -> 记录日志
```

`optimizer.zero_grad()` 很重要，因为 PyTorch 默认会累积梯度。如果不清空，上一个 batch 的梯度会混到下一个 batch。AMP 的作用是让部分计算用低精度，提高速度并节省显存；但为了避免梯度下溢，fp16 训练常配合 GradScaler。

面试中能手写训练循环，并解释每一行在做什么，基本就证明你真正跑过模型。

### 4.3 Dataset 和 DataLoader

常见瓶颈：

- CPU 数据预处理慢。
- 磁盘 IO 慢。
- 小文件过多。
- 数据增强过重。
- `num_workers` 设置不合理。
- GPU 等数据，利用率低。

优化手段：

- 合理设置 `num_workers`、`pin_memory`、`prefetch_factor`。
- 缓存预处理结果。
- 使用 WebDataset / parquet / mmap / LMDB。
- 把部分增强搬到 GPU。
- 训练时监控 data time 和 compute time。

高频问题答案：

**`num_workers` 怎么调？** 它控制 DataLoader 用多少子进程加载数据。太小 GPU 等数据，太大可能 CPU 争抢、内存暴涨或 Windows 下启动开销大。通常从 0、2、4、8 逐步测。

**`pin_memory` 有什么用？** 它把 CPU 内存页锁定，提升 CPU 到 GPU 的异步拷贝效率，CUDA 训练中常开启。

**为什么小文件多会慢？** 每个文件打开、寻址、解码都有额外开销。可以考虑打包成 LMDB、WebDataset、parquet 或缓存预处理结果。

**怎么判断瓶颈在数据还是模型？** 记录每个 batch 的 data time 和 compute time。GPU 利用率低且 data time 高，优先优化数据管道。

从 0 理解：

Dataset 负责“知道样本在哪里、怎么读、怎么变成 tensor”；DataLoader 负责“按 batch 取数据，并行加载，打乱顺序”。很多新手只关注模型，却忽略数据加载，结果 GPU 一直在等 CPU。

一个样本从磁盘到模型通常经历：

```text
读取文件 -> 解码/解析 -> 预处理 -> 数据增强 -> 转 tensor -> 组成 batch -> 拷贝到 GPU
```

如果训练慢，要分别统计 data time 和 compute time。data time 高，说明数据管道慢；compute time 高，才是模型计算慢。这个区分在工程面试里很加分。

### 4.4 Debug 能力

训练不收敛排查清单：

- loss 是否真的在下降？
- label 是否错位？
- train/eval 模式是否正确？
- 学习率是否过大或过小？
- 输入是否 normalize？
- loss 和 logits shape 是否匹配？
- 是否有 NaN / Inf？
- 数据划分是否泄漏？
- baseline 是否可学到简单模式？

显存 OOM 排查：

- 降低 batch size。
- 使用 AMP。
- gradient checkpointing。
- 清理无用引用。
- 避免在 list 中保存带计算图的 tensor。
- 使用 profiler / memory snapshot。
- 检查评估时是否用了 `torch.no_grad()`。

从 0 理解：

Debug 的核心是先缩小问题。不要一上来改模型结构。训练不收敛时，先拿很小的数据集做 overfit test：比如只取 10 个样本，看模型能不能把训练 loss 降到很低。如果连 10 个样本都记不住，通常是代码、标签、loss、学习率或 shape 有问题。

推荐排查顺序：

```text
数据是否正确 -> label 是否对齐 -> forward shape 是否正确 -> loss 是否合理 -> 梯度是否存在 -> 参数是否更新 -> 验证逻辑是否正确
```

OOM 也要分清原因：是模型参数太大、activation 太大、batch 太大、序列太长，还是代码保存了不该保存的计算图。能说清这些，面试官会知道你不是只会“batch size 调小”。

高频问题答案：

**训练 loss 不下降怎么办？** 先用极小数据集过拟合测试。如果过拟合不了，检查 label、loss、shape、学习率、梯度和参数是否更新；如果能过拟合但验证差，说明泛化或数据分布有问题。

**出现 NaN 怎么排查？** 检查输入是否有 NaN/Inf，学习率是否太大，loss 是否数值不稳定，混合精度是否溢出，梯度是否爆炸。必要时关闭 AMP 定位。

**显存一直涨怎么办？** 常见原因是在 list 中保存了带计算图的 tensor，或评估时没用 `no_grad()`。保存日志时用 `.item()` 或 `.detach().cpu()`。

### 4.5 PyTorch 自定义算子

适合自定义算子的情况：

- PyTorch 原生算子组合产生大量中间 tensor。
- 某段计算频繁调用，kernel launch overhead 高。
- 需要 fuse 多个操作减少显存读写。
- 需要利用特定数据布局或硬件特性。

这些情况具体是什么意思：

- **大量中间 tensor**：例如 `bias -> gelu -> dropout -> residual` 每一步都生成中间结果，显存读写多。融合后中间值可以不落显存。
- **kernel launch overhead 高**：很多小算子每次启动 GPU kernel 都有固定开销，算子本身很小的时候启动开销占比很高。
- **fuse 多个操作**：把多个逐元素操作、归一化或 reduction 合到一个 kernel，减少 global memory 往返。
- **特殊数据布局**：某些业务数据不是 PyTorch 默认 contiguous layout，手写 kernel 可以直接按目标布局访问，避免额外转换。

基本路径：

1. Python 写 reference 实现。
2. 写 C++/CUDA extension。
3. forward 对齐 PyTorch 结果。
4. 如果训练需要，补 backward 或用 autograd 包装。
5. 用 `torch.utils.benchmark` 或 CUDA event 计时。
6. 用 Nsight Systems / Nsight Compute 分析瓶颈。

高频问题答案：

**什么时候值得写自定义算子？** 当原生 PyTorch 组合产生大量中间 tensor、kernel launch 很多、访存来回太多，或者需要特殊数据布局时值得考虑。否则维护成本可能超过收益。

**怎么验证自定义算子正确？** 先写 PyTorch reference，对比输出最大绝对误差和相对误差；训练相关算子还要做 backward 和 gradcheck。

**怎么 benchmark？** 先 warmup，再用 CUDA event 或 `torch.utils.benchmark` 计时；要同步 CUDA；要比较不同 shape、dtype、batch，并报告误差和耗时。

从 0 理解：

PyTorch 自带很多算子，但有时多个小算子串起来会产生大量中间 tensor 和 kernel launch。例如 `x -> bias -> gelu -> dropout -> residual` 如果分开执行，每一步都可能读写显存。自定义算子可以把这些操作融合，减少显存读写和启动开销。

自定义算子开发不是一开始就追求最快，而是按闭环做：

```text
PyTorch reference -> 自定义 forward -> 正确性对齐 -> benchmark -> profiler -> 优化 -> backward/训练支持
```

面试时最重要的是说清“为什么要自定义”。如果原生算子已经很快，盲目手写 CUDA 可能没有收益。真正合理的理由通常是融合、特殊数据布局、减少中间结果、支持框架没有的算子或满足业务端到端延迟要求。

## 5. CUDA 和算子开发

这一章按 BasicCUDA 的思路：用可运行的小例子逐步理解，而不是一上来背 GPU 架构。

从 0 理解 CUDA，要先明白为什么需要 GPU。CPU 像少量很强的工人，适合复杂控制逻辑；GPU 像大量相对简单的工人，适合把同一种计算同时应用到海量数据上。深度学习中大量操作都是矩阵乘、向量加、归一化、softmax，这些计算规则相似、数据量巨大，所以非常适合 GPU。

算子就是一个基本计算单元，例如 matmul、softmax、layernorm、relu。模型前向不是一个整体黑盒，而是很多算子串起来：

```text
输入 tensor -> matmul -> add bias -> activation -> norm -> attention -> ...
```

算子开发的目标是让这些基本计算在特定硬件上更快、更省显存、更稳定。学习顺序建议：

```text
向量加法 -> reduction -> 矩阵乘 -> shared memory -> softmax/layernorm -> PyTorch extension -> Triton/NCCL
```

面试中不要只说“我会 CUDA”，要能讲清一个 kernel 的线程如何映射到数据、数据从 global memory 到 register/shared memory 怎么流动、性能瓶颈如何被 profiler 证实。

### 5.1 CUDA 编程模型

层级：

- Grid：一次 kernel launch 的线程块集合。
- Block：线程块，内部线程可以共享 shared memory，并能 `__syncthreads()`。
- Thread：执行最小单元。
- Warp：通常 32 个线程，是 GPU 调度的基本执行单位。

层级怎么实际使用：

- **Grid 怎么设**：grid 负责覆盖整个输入数据。例如处理 N 个元素时，grid 中所有 block 的线程总数要足够覆盖 N，或者用 grid-stride loop 重复处理。
- **Block 怎么设**：block 是调度和共享内存的单位。常见 block size 是 128、256、512 线程，需要结合 occupancy、寄存器、shared memory 调。
- **Thread 怎么设**：thread 负责最小粒度计算，例如一个元素、一个输出位置或一个小向量。设计 thread 映射时要保证访存尽量连续。
- **Warp 怎么考虑**：warp 内 32 个线程一起执行。优化时要减少 warp divergence，并尽量让 warp 内线程访问连续地址。

常用索引：

```cpp
int tid = blockIdx.x * blockDim.x + threadIdx.x;
int stride = blockDim.x * gridDim.x;
for (int i = tid; i < n; i += stride) {
    y[i] = x[i] * scale;
}
```

为什么 grid-stride loop 重要：

- kernel 可以处理任意长度。
- 可以复用线程。
- 避免启动过多 block。

高频问题答案：

**Grid、Block、Thread 是什么关系？** Grid 是一次 kernel 启动的所有线程块集合，Block 是一组可以共享 shared memory 的线程，Thread 是执行具体计算的最小单位。常见设计是“一个 thread 处理一个元素”或“一个 block 处理一个 tile/一行数据”。

**Warp 是什么？** Warp 通常是 32 个线程，是 GPU 调度和执行的基本单位。同一个 warp 内线程尽量执行同一条指令，所以分支发散会降低效率。

**为什么要做边界检查？** 数据长度不一定刚好等于线程数整数倍，最后一些线程可能映射到不存在的元素。`if (idx < n)` 可以避免越界访问。

从 0 理解：

一次 CUDA kernel launch 就像给 GPU 发布一个任务。这个任务被拆成很多 block，每个 block 里有很多 thread。每个 thread 通常负责处理一小份数据，比如向量加法中一个 thread 负责一个元素。

为什么要有 block？因为 GPU 需要把大量线程组织起来调度；同一个 block 内的线程可以共享 shared memory，也可以同步。不同 block 之间通常不能直接同步，所以设计算法时要想清楚哪些数据需要 block 内合作，哪些需要分多阶段 kernel。

线程映射是算子开发第一步。例如矩阵 C 有 `M*N` 个元素，你可以让每个线程计算一个元素；softmax 有很多行，你可以让一个 block 处理一行。映射方式决定并行度、访存模式和后续优化空间。

### 5.2 访存层级

从快到慢粗略理解：

- Register：每个线程私有，最快，但数量有限。
- Shared memory：block 内共享，速度快，可手动管理。
- L1 / L2 cache：硬件缓存。
- Global memory：显存，容量大但延迟高。
- Host memory：CPU 内存，跨 PCIe / NVLink 访问更慢。

访存层级怎么实际使用：

- **Register 怎么用**：线程内部临时变量默认可能放在寄存器。高性能 matmul 会用 register 保存局部累加结果。
- **Shared memory 怎么用**：把一个 tile 从 global memory 搬进 shared memory，让 block 内线程反复复用。典型例子是 tiled matmul 和 block-level reduction。
- **L1/L2 cache 怎么用**：一般由硬件管理，但你的访问模式会影响命中率。连续、重复访问更容易命中。
- **Global memory 怎么用**：容量大但慢，优化目标是少访问、连续访问、合并访问。
- **Host memory 怎么用**：CPU 内存到 GPU 显存拷贝很慢，训练时要减少频繁小拷贝，必要时用 pinned memory 和异步拷贝。

优化核心：

- 减少 global memory 访问次数。
- 保证 coalesced memory access。
- 提高数据复用。
- 避免 bank conflict。
- 避免不必要同步。
- 控制 register 使用，防止 occupancy 下降。

高频问题答案：

**什么是 coalesced memory access？** 相邻线程访问相邻内存地址，GPU 可以把多个访问合并成更少的内存事务，带宽利用率更高。反过来，线程访问地址很分散，就会浪费带宽。

**shared memory 为什么快？** 它在 SM 内部，延迟远低于 global memory，并且可以被同一个 block 的线程共享。适合存放会被重复使用的小块数据。

**register 用越多越好吗？** 不是。register 很快，但每个 SM 资源有限。单线程 register 用太多，会减少同时驻留的线程数量，降低 occupancy。

从 0 理解：

GPU 算得很快，但从显存拿数据相对慢。很多 kernel 并不是“算不动”，而是“等数据”。所以 CUDA 优化经常围绕访存：少读、连续读、重复利用、别乱读。

可以用厨房类比：global memory 像仓库，容量大但拿东西慢；shared memory 像操作台，容量小但近；register 像手里拿着的工具，最快但很有限。高性能 kernel 会尽量把常用数据搬到更近的位置，并让多个线程复用。

coalesced access 的直觉是：相邻线程访问相邻地址，硬件可以合并成更高效的内存事务。如果线程 0 读地址 0、线程 1 读地址 1000、线程 2 读地址 7，访问会很低效。

### 5.3 性能分析基本指标

关键指标：

- Kernel time。
- Achieved occupancy。
- SM utilization。
- Memory throughput。
- L2 hit rate。
- Warp stall reasons。
- Tensor Core utilization。

指标怎么看：

- **Kernel time**：单个 kernel 耗时，直接看优化是否变快。
- **Achieved occupancy**：实际活跃 warp 占理论上限的比例。低 occupancy 可能无法隐藏延迟，但高 occupancy 不保证最快。
- **SM utilization**：SM 是否忙。低利用率可能是 CPU 调度、数据拷贝、batch 太小或 kernel 太碎。
- **Memory throughput**：显存带宽使用情况。接近硬件上限时说明很可能 memory-bound。
- **L2 hit rate**：L2 缓存命中率。命中率低可能说明访问模式差或数据复用不足。
- **Warp stall reasons**：warp 为什么停住，例如等内存、等同步、等依赖。它能指导优化方向。
- **Tensor Core utilization**：Tensor Core 是否被用起来。matmul/conv 低利用时要检查 dtype、shape 对齐和调用路径。

Roofline 思维：

- 算术强度 = FLOPs / bytes。
- 算术强度低：memory-bound，优化访存。
- 算术强度高：compute-bound，优化计算流水、Tensor Core、并行度。

面试回答：

“我不会先猜优化方向，而是先拿 baseline 跑 profiler，看瓶颈是访存、计算、同步还是 launch overhead。然后用 roofline 判断理论上限，再逐步验证每个优化是否真的带来收益。”

高频问题答案：

**Nsight Systems 和 Nsight Compute 区别？** Nsight Systems 看端到端时间线，例如 CPU 调度、kernel launch、拷贝、通信和等待；Nsight Compute 深入看单个 kernel 的硬件指标，例如 occupancy、访存吞吐、warp stall、cache 命中率。

**什么是 memory-bound？** 主要时间花在读写内存，计算单元没有被喂饱。优化方向是减少访存、合并访存、提高复用、做 fusion。

**什么是 compute-bound？** 主要时间花在计算，内存不是主要瓶颈。优化方向是提高并行度、使用 Tensor Core、减少低效指令、改善数据布局。

从 0 理解：

优化前必须先测量。否则你可能花很多时间优化一个不是瓶颈的地方。CUDA 性能分析要回答：

```text
慢在哪里 -> 为什么慢 -> 理论上能多快 -> 改动后是否真的变快
```

memory-bound 的意思是主要时间花在读写数据；compute-bound 的意思是主要时间花在计算。向量加法通常 memory-bound，因为每个元素只做一次加法，却要读两个数写一个数；大矩阵乘更可能 compute-bound，因为每读一块数据可以做很多乘加。

Occupancy 不是越高越好。更高 occupancy 可能隐藏延迟，但如果为了提高 occupancy 牺牲了数据复用或增加了访存，反而变慢。面试中能说出这一点，会显得很成熟。

### 5.4 从向量加法到矩阵乘

向量加法学习目标：

- kernel launch。
- thread/block/grid 映射。
- 边界检查。
- host-device memory copy。
- CUDA error check。

矩阵乘学习目标：

- 朴素实现：每个线程算一个 C 元素。
- 1D mapping：线程线性映射到 C。
- 2D mapping：block 和 thread 对应矩阵 tile。
- shared memory tiling：复用 A 和 B 的子块。
- 任意尺寸支持：处理边界。
- 对比 cuBLAS：理解自己实现和工业库差距。

怎么做：

1. 先写 CPU 或 PyTorch reference，确定正确结果。
2. 写 naive CUDA：每个 thread 计算一个 `C[row, col]`。
3. 加边界判断，支持任意 `M, N, K`。
4. 用 CUDA event 计时，和 PyTorch/cuBLAS 比较。
5. 再做 shared memory tiling，减少 A/B 重复 global load。
6. 最后考虑 vectorized load、register tiling、Tensor Core。

高频问题答案：

**Matmul 为什么适合 tiling？** 因为 A 的一个元素会被同一行多个输出复用，B 的一个元素会被同一列多个输出复用。把 A/B 子块放进 shared memory，可以让 block 内线程重复使用，减少 global memory 访问。

**为什么自己写的 matmul 很难超过 cuBLAS？** cuBLAS 做了大量硬件级优化，包括 Tensor Core、寄存器分块、流水线、数据布局、autotune 和多架构适配。自写 kernel 的价值在学习优化逻辑或做特殊融合，不一定要超过工业库。

复杂度：

```text
C[M, N] = A[M, K] * B[K, N]
FLOPs ≈ 2 * M * N * K
```

优化直觉：

- 朴素 kernel 每次从 global memory 读 A、B，重复读很多次。
- tiled matmul 把 A、B 子块放入 shared memory，block 内复用。
- 再进一步要考虑 register tiling、vectorized load、双缓冲、Tensor Core。

从 0 理解：

向量加法是 CUDA 的 hello world，因为每个输出元素互不依赖：

```text
C[i] = A[i] + B[i]
```

矩阵乘就更接近深度学习核心：

```text
C 的一个元素 = A 的一行 和 B 的一列 做点积
```

朴素实现中，每个线程算一个 C 元素，会反复从 global memory 读取 A 和 B。shared memory tiling 的想法是：一个 block 负责 C 的一个小块，把对应的 A 子块和 B 子块先搬到 shared memory，block 内线程反复使用这些数据。这样减少 global memory 访问，通常能明显提速。

BasicCUDA 风格的学习重点不是一次写出 cuBLAS，而是每个版本都回答：

- 正确性是否对齐？
- 比上一个版本快在哪里？
- 多用了哪些硬件资源？
- 哪些尺寸下反而不快？
- 和 cuBLAS 的差距来自哪里？

### 5.5 Shared Memory

适用条件：

- 同一块数据会被 block 内多个线程重复使用。
- 访问模式可预测。
- shared memory 占用不会严重降低 occupancy。

典型坑：

- 加 shared memory 不一定更快。
- 如果数据复用不足，同步和搬运成本可能超过收益。
- bank conflict 会拖慢访问。
- 动态 shared memory 大小要谨慎。

高频问题答案：

**bank conflict 是什么？** shared memory 被分成多个 bank。如果同一个 warp 中多个线程访问同一个 bank 的不同地址，就会串行化，变慢。优化方式包括调整数据布局、padding 或改变访问模式。

**`__syncthreads()` 有什么限制？** 它只能同步同一个 block 内线程，不能跨 block；而且必须保证 block 内所有线程都能执行到同一个同步点，否则可能死锁。

**什么时候 shared memory 反而变慢？** 数据复用不足、同步太多、bank conflict 严重或 shared memory 占用过高导致 occupancy 降低时，可能不如直接读 global memory。

面试可讲：

“我会先估算数据复用次数。比如矩阵乘中，一个 tile 的 A 和 B 会被多个线程重复使用，因此 shared memory tiling 收益明显；但如果每个元素只用一次，把它搬到 shared memory 反而可能变慢。”

从 0 理解：

shared memory 不是“自动加速按钮”，它是一个需要你手动管理的小缓存。使用它通常有三个步骤：

```text
线程协作从 global memory 读取数据 -> 放入 shared memory -> 同步 -> 多次复用 -> 写回结果
```

为什么需要 `__syncthreads()`？因为 block 内线程是并行执行的。某些线程可能还没把数据写进 shared memory，其他线程就开始读，会读到错误结果。同步保证大家都搬完数据再继续。

但同步也有成本。如果数据只用一次，搬进 shared memory 再同步反而多此一举。所以判断 shared memory 是否有用，核心看“复用次数”和“访存模式”。

### 5.6 Warp 和分支

要点：

- 一个 warp 内线程执行同一指令。
- 如果 warp 内分支不同，会发生 divergence，分支路径串行执行。
- 边界判断不可避免，但要尽量让主路径规整。

常见优化：

- 用 warp-level primitive 做 reduce / scan。
- 减少 warp 内不一致分支。
- 让连续线程访问连续内存。

高频问题答案：

**分支发散为什么慢？** 同一 warp 内线程执行不同分支时，GPU 需要分别执行每条分支路径，再屏蔽不走该路径的线程，等于部分线程空转。

**warp-level primitive 有什么用？** 它允许 warp 内线程直接交换数据或做 reduce，不一定要通过 shared memory，常用于高效 reduction、scan、softmax 等。

从 0 理解：

GPU 不是每个线程完全独立乱跑，而是以 warp 为单位执行。一个 warp 通常 32 个线程，它们尽量同时执行同一条指令。如果同一个 warp 里一半线程走 if 分支，一半走 else 分支，GPU 可能要分两次执行，这就是分支发散。

边界判断不可避免，比如处理长度不是 block size 整数倍的数据。但核心计算区域应尽量规整，让多数线程做同样的事、访问连续地址。

### 5.7 原子操作和 reduction

Reduction 常见实现：

- 每个 block 先在 shared memory 内 reduce。
- block 结果写到全局内存。
- 再二阶段 reduce 或用 atomic。

高频问题答案：

**atomic 为什么可能慢？** 如果很多线程同时对同一个地址做 atomic，加法会串行化，形成热点。少量 atomic 可以接受，大规模 reduction 通常要分层聚合。

**浮点 reduction 为什么结果可能不完全一样？** 浮点加法不满足严格结合律，不同并行顺序会导致舍入误差不同。通常比较时用误差容忍，而不是要求 bitwise 完全一致。

注意：

- atomic 简单但可能成为热点。
- 浮点 reduction 不满足严格结合律，不同顺序结果可能略有差异。
- 面试中要主动提数值稳定性和确定性。

从 0 理解：

Reduction 是“把很多数合成一个数”，例如求和、求最大值。它在 softmax、layernorm、loss 统计中非常常见。难点是很多线程同时参与计算，不能让大家随便写同一个地址，否则会冲突。

atomic 可以保证多个线程写同一个位置时不互相覆盖，但如果太多线程都抢一个位置，就会变慢。更常见的高性能做法是分层 reduction：先在 block 内求局部结果，再把局部结果合并。

浮点数求和顺序不同，结果可能有微小差异。这不是 bug，而是浮点数精度特性。训练和推理中要知道容忍误差范围。

### 5.8 算子融合

为什么融合：

- 减少 kernel launch 次数。
- 减少中间 tensor 读写 global memory。
- 提高缓存局部性。

例子：

```text
y = dropout(gelu(x @ w + b))
```

如果拆成多个 kernel，中间结果会多次写回显存；融合后可以在寄存器或 shared memory 中完成部分计算。

典型 fused softmax：

1. 每行求 max，保证数值稳定。
2. 每行计算 exp(x - max)。
3. 求 sum。
4. 除以 sum。

高频问题答案：

**Softmax 为什么要减最大值？** 因为 `exp(x)` 对大数容易溢出。减去最大值不改变 softmax 结果，但能把最大指数变成 `exp(0)=1`，保证数值稳定。

**为什么 kernel fusion 能提升性能？** 多个小算子拆开会多次读写 global memory 和多次 kernel launch。融合后中间结果可以保存在 register/shared memory 中，减少访存和启动开销。

**fusion 有什么风险？** 代码复杂、通用性下降、寄存器压力增加、occupancy 下降，并且 backward 也可能需要配套实现。必须 benchmark 验证。

优化点：

- 一行对应一个 block 或多个 warp。
- reduce max 和 reduce sum。
- 尽量减少 global memory 往返。
- 注意 mask 和 causal mask。
- backward 也要考虑融合。

从 0 理解：

算子融合的核心不是“少写几行代码”，而是减少数据在显存里来回搬。GPU 计算很快，但显存读写贵。如果一个中间结果只为了传给下一个算子，就可以考虑把两个算子合在一个 kernel 里，让中间值留在 register 或 shared memory。

以 softmax 为例，不能直接算 `exp(x)`，因为 x 很大时会溢出。所以稳定 softmax 会先减去每行最大值：

```text
softmax(x_i) = exp(x_i - max(x)) / sum_j exp(x_j - max(x))
```

这个过程包含求最大值、求指数、求和、归一化，本质上是几次 reduction 和逐元素计算。fused softmax 要把这些步骤尽量放在一个 kernel 中完成，并处理 mask、行长度、dtype 和 backward。

### 5.9 PyTorch CUDA Extension

最小知识栈：

- C++ binding：pybind11 / TORCH_LIBRARY。
- CUDA kernel：`.cu`。
- build：`setup.py` 或 `torch.utils.cpp_extension.load`。
- tensor 检查：device、dtype、contiguous、shape。
- stream：使用当前 PyTorch CUDA stream。

高频问题答案：

**为什么要检查 contiguous？** 自定义 kernel 往往假设内存连续。如果传入 non-contiguous tensor，按线性地址读取会错。可以要求 `.contiguous()`，或在 kernel 中支持 stride。

**为什么要使用当前 PyTorch stream？** PyTorch 可能在不同 stream 上调度算子。自定义 kernel 如果不跟随当前 stream，可能出现同步错误或性能问题。

**forward 对齐就够了吗？** 如果只做推理可能够；如果要训练，还需要 backward、梯度正确性测试和 autograd 集成。

面试可讲的完整闭环：

1. 用 Python reference 保证语义。
2. 写 CUDA forward。
3. 和 PyTorch 结果比较 `max_abs_error`、`max_rel_error`。
4. 写 benchmark，区分 warmup 和正式计时。
5. 用 profiler 看瓶颈。
6. 如果训练需要，写 backward 并做 gradcheck。

从 0 理解：

PyTorch CUDA Extension 是把自定义 CUDA kernel 接到 PyTorch 生态中的方式。这样你可以像调用普通 PyTorch 函数一样调用自己的 kernel，并和模型训练、autograd、benchmark 工具结合。

最小可用工程应包含：

```text
Python 测试脚本
C++ binding
CUDA kernel
构建脚本
正确性测试
性能测试
```

面试中要强调工程闭环：只写 kernel 不够，还要检查输入 tensor 是否在 CUDA 上、是否 contiguous、dtype 是否支持、shape 是否符合预期，以及 kernel 是否使用当前 PyTorch stream，避免和框架调度冲突。

### 5.10 Triton

Triton 的定位：

- 用 Python 风格写 GPU kernel。
- 比 CUDA C++ 更容易写和调试。
- 适合快速实现 matmul、softmax、layernorm、activation fusion 等。

需要掌握：

- `@triton.jit`。
- program id。
- block size。
- mask load/store。
- `tl.arange`、`tl.load`、`tl.store`。
- autotune。

高频问题答案：

**Triton 和 CUDA 怎么选？** 快速写深度学习算子、做实验和自动调参时优先 Triton；需要极致控制、复杂同步、特殊硬件特性或工程集成时用 CUDA。

**Triton 的 mask 是做什么的？** block/tile 可能覆盖到越界位置，mask 用来控制哪些元素真正 load/store，避免越界。

**autotune 是什么？** 自动尝试不同 block size、num warps 等配置，选择当前 shape 和硬件上更快的版本。

CUDA 和 Triton 的取舍：

- CUDA：控制力强，适合极致优化和复杂底层集成。
- Triton：开发效率高，适合快速实验和许多深度学习算子。
- 面试中可以说：先用 Triton 验证优化思路，性能或特殊控制不足时再用 CUDA。

从 0 理解：

Triton 可以理解成“更接近 Python 的 GPU kernel 编写方式”。它隐藏了一部分 CUDA 细节，但仍然需要你理解 block、mask、访存和并行。很多深度学习算子用 Triton 写起来比 CUDA C++ 短很多，适合快速迭代。

Triton 的 program 类似 CUDA 的 block。每个 program 处理一个 tile，通过 `tl.load` 读取数据，通过 mask 处理越界，通过 `tl.store` 写回结果。它很适合 matmul、softmax、layernorm 这类规则张量计算。

学习建议：同一个算子先写 PyTorch reference，再写 Triton，再和 CUDA 或 PyTorch 原生算子比较。这样你会理解“语义正确”和“性能优秀”是两层不同目标。

### 5.11 NCCL 和多 GPU 通信

必须理解的 collective：

- AllReduce：各卡梯度求和再广播。
- Broadcast：一张卡发给所有卡。
- ReduceScatter：reduce 后切分给各卡。
- AllGather：收集各卡分片。
- AllToAll：各卡互相交换数据，MoE 常见。

collective 怎么实际发生：

- **AllReduce 怎么做**：每张卡都有一份 tensor，通信后每张卡都得到所有卡求和/平均后的结果。DDP 梯度同步用它。
- **Broadcast 怎么做**：一个 rank 作为源，把参数或配置发给其他 rank。初始化模型权重时常见。
- **ReduceScatter 怎么做**：先把多卡数据 reduce，再把结果分片给不同卡。ZeRO/FSDP 用它减少每卡保存的数据。
- **AllGather 怎么做**：每张卡拿着一片数据，通信后每张卡都拿到完整数据。FSDP 前向需要完整参数时会用。
- **AllToAll 怎么做**：每张卡给每张卡发送不同数据。MoE 把 token 路由到不同 expert 时常见。

高频问题答案：

**AllReduce 用在哪里？** DDP 中每张卡计算一份梯度，AllReduce 把所有卡的梯度求和/平均，让每张卡得到一致梯度。

**ReduceScatter 和 AllGather 用在哪里？** FSDP/ZeRO 中参数、梯度、优化器状态被切分。ReduceScatter 用于把聚合结果分发给各卡，AllGather 用于需要完整参数时临时收集。

**通信为什么会拖慢训练？** GPU 计算很快，但跨卡/跨机传输带宽和延迟有限。如果通信不能和 backward 重叠，就会让 GPU 等网络。

训练中的关系：

- DDP 梯度同步常用 AllReduce。
- FSDP / ZeRO 会用 ReduceScatter 和 AllGather 管理参数、梯度、优化器状态。
- Tensor Parallel 需要频繁跨卡通信。

性能关注：

- 拓扑：PCIe、NVLink、IB、RoCE。
- 消息大小：小消息 latency-bound，大消息 bandwidth-bound。
- overlap：通信和计算重叠。
- bucket size：影响同步粒度。

从 0 理解：

单 GPU 装不下或算不快时，就需要多 GPU。多 GPU 的难点不是每张卡自己算，而是卡与卡之间要交换数据。NCCL 提供了高性能 collective 通信，让训练框架可以做梯度同步、参数聚合、分片收集。

以 DDP 为例：每张 GPU 拿不同 batch，算出各自梯度。为了让模型参数保持一致，需要把所有卡的梯度求平均，这就是 AllReduce。AllReduce 太慢，训练就会卡在通信上。

AI Infra 面试常问通信和计算 overlap：如果某些层的梯度已经算完，可以先启动这些梯度的通信，同时继续算前面层的梯度，从而隐藏一部分通信时间。

### 5.12 算子开发面试题

高频问题：

- 如何实现一个高性能 LayerNorm？
- Softmax 为什么要减最大值？
- Matmul 怎么用 shared memory 优化？
- 为什么 kernel fusion 能提升性能？
- 如何判断一个 kernel 是 memory-bound 还是 compute-bound？
- Occupancy 越高一定越好吗？
- CUDA stream 和 event 有什么用？
- `__syncthreads()` 有什么限制？
- fp16 累加为什么可能要用 fp32？
- Tensor Core 适合什么数据类型和矩阵尺寸？

高频问题答案：

**如何判断 kernel 是 memory-bound 还是 compute-bound？** 看算术强度、显存带宽利用率、SM 利用率和 profiler stall reason。如果带宽接近上限而计算单元利用不高，多半 memory-bound；如果 Tensor Core/ALU 忙而带宽不是瓶颈，多半 compute-bound。

**Occupancy 越高一定越好吗？** 不一定。Occupancy 高可以隐藏延迟，但如果为了提高 occupancy 牺牲数据复用、增加访存或降低单线程效率，整体可能更慢。

**CUDA stream 和 event 有什么用？** stream 用于组织异步执行和重叠拷贝/计算；event 用于精确计时和跨 stream 同步。

**fp16 累加为什么可能用 fp32？** fp16 范围和精度有限，长 reduction 或 matmul 累加误差会变大，甚至溢出。常见做法是 fp16 输入、fp32 accumulate。

**Tensor Core 适合什么？** 适合特定 dtype 和 shape 的矩阵乘/卷积，例如 fp16/bf16/tf32/int8。要发挥性能，矩阵维度和内存布局通常要满足硬件友好的对齐要求。

回答 LayerNorm 的结构：

1. 输入 shape 通常是 `[B, H]`，对 H 维做归一化。
2. 每行求 mean 和 variance。
3. 输出 `(x - mean) / sqrt(var + eps) * gamma + beta`。
4. 性能瓶颈通常是多次读写和 reduction。
5. 优化方式：每行一个 block、向量化 load、warp/block reduce、融合 affine、减少中间写。

从 0 到可面试的算子项目建议：

1. 写一个 vector add，理解 kernel launch 和线程索引。
2. 写一个 reduce sum，理解 shared memory 和同步。
3. 写一个 naive matmul，理解 2D 映射。
4. 写一个 tiled matmul，理解数据复用。
5. 写一个 softmax 或 layernorm，理解 reduction、数值稳定和融合。
6. 接入 PyTorch extension 或 Triton，做 correctness + benchmark。

面试讲项目时按这个顺序讲：

```text
原始实现 -> 发现瓶颈 -> 优化假设 -> 具体改动 -> profiler/benchmark 结果 -> 边界和不足
```

这比只说“我实现了某个 kernel”更有说服力。

## 6. AI Infra

AI Infra 是把 AI 模型变成可用系统的工程层。训练一个 notebook demo 和上线一个稳定 AI 服务，是完全不同的难度。AI Infra 关注的问题包括：数据怎么流动，模型怎么训练，训练失败怎么恢复，推理请求怎么排队，GPU 怎么利用，结果怎么监控，成本怎么控制。

从 0 理解，可以把 AI Infra 看成三条主线：

```text
训练基础设施：让模型高效、可复现地训练出来
推理基础设施：让模型低延迟、高吞吐地服务用户
平台基础设施：让数据、模型、日志、监控、权限和版本可管理
```

面试中如果能把算法问题放进系统链路里讲，就会明显区分于只会调包的候选人。

### 6.1 AI 系统全链路

一个真实 AI 系统通常包括：

- 数据采集。
- 数据清洗和标注。
- 训练数据版本管理。
- 模型训练。
- 实验追踪。
- 模型评估。
- 模型注册。
- 部署服务。
- 监控和告警。
- 反馈闭环。

这些环节分别做什么：

- **数据采集**：定义数据从哪里来，例如业务日志、DICOM、用户反馈、公开数据集。
- **数据清洗和标注**：去重、脱敏、修正错误、统一标签标准。标注质量决定模型上限。
- **训练数据版本管理**：记录每次训练用了哪批数据，避免“模型变了但不知道数据也变了”。
- **模型训练**：用训练框架跑实验，保存配置、日志、checkpoint。
- **实验追踪**：记录每次实验参数和指标，支持对比和复现。
- **模型评估**：不仅看离线指标，也看分桶、鲁棒性、安全性、延迟和成本。
- **模型注册**：把可上线模型登记入库，记录版本、指标、文件地址和审批状态。
- **部署服务**：把模型包装成 API、批处理任务或边缘端组件。
- **监控告警**：监控延迟、错误率、资源、输入分布和质量反馈。
- **反馈闭环**：把线上错误和用户反馈变成下一轮数据或规则改进。

面试时要能画出链路：

```text
Data -> Feature/Preprocess -> Train -> Evaluate -> Registry -> Serve -> Monitor -> Feedback
```

怎么做：

1. 数据层：定义数据来源、schema、标注和版本。
2. 训练层：记录代码、配置、数据版本和 checkpoint。
3. 评估层：离线指标、人工评估、压力测试都要有。
4. 发布层：模型注册、灰度、回滚、权限控制。
5. 监控层：延迟、错误率、资源、质量反馈。

高频问题答案：

**为什么 AI 系统需要闭环？** 因为模型上线后会遇到训练集中没有的数据，业务也会变化。没有反馈闭环，就无法发现错误样本、分布漂移和用户真实痛点。

**模型注册表有什么用？** 它把模型文件、训练数据、代码版本、指标和上线状态绑定，保证能追溯、回滚和比较版本。

从 0 理解：

一个 AI 系统不是“训练完模型就结束”。数据会变，业务会变，模型会上线出错，用户会提出新需求。因此需要闭环。模型训练只是中间一环，前面有数据治理，后面有部署、监控和反馈。

模型注册表的作用是记录“哪个模型版本可以上线”，包括训练数据、代码版本、指标和文件位置。监控的作用是发现线上是否变慢、变错、变贵或数据分布漂移。反馈闭环则把真实使用中的错误样本收集回来，变成下一轮训练数据。

### 6.2 分布式训练

主要并行方式：

- Data Parallel：每张卡一份模型，不同数据，梯度同步。
- Tensor Parallel：模型单层矩阵按维度切分。
- Pipeline Parallel：不同层放不同设备，流水线执行。
- Sequence Parallel：序列维度切分，常用于 Transformer。
- Expert Parallel：MoE 中不同 expert 分布到不同设备。

并行方式怎么选：

- **Data Parallel 怎么做**：每张 GPU 放完整模型，吃不同 batch，反向后同步梯度。模型能放进单卡、想提升吞吐时优先用。
- **Tensor Parallel 怎么做**：把一个大矩阵乘拆到多张卡，例如按列或按行切 Linear 权重。单层太大或想训练超大模型时用。
- **Pipeline Parallel 怎么做**：把不同层放到不同 GPU，micro-batch 像流水线一样通过。模型层数很多、单卡放不下时用。
- **Sequence Parallel 怎么做**：把序列维度切开，减少某些激活和归一化的显存压力。通常和 tensor parallel 搭配。
- **Expert Parallel 怎么做**：MoE 中不同 expert 放不同卡，token 根据 router 分发到 expert。难点是 AllToAll 通信和负载均衡。

DDP：

- 每个进程一张 GPU。
- forward/backward 本地计算。
- backward 时梯度 bucket 触发 AllReduce。
- 每张卡得到同步后的梯度并更新。

FSDP / ZeRO：

- 把参数、梯度、优化器状态切分到不同 GPU。
- 降低单卡显存。
- 代价是通信更复杂。

常见问题：

- 为什么 batch size 扩大后需要调学习率？
- 梯度累积和数据并行有什么区别？
- 通信如何和计算 overlap？
- checkpoint 如何保存和恢复？
- 多机训练失败如何容错？

高频问题答案：

**梯度累积和数据并行区别是什么？** 梯度累积是在一张卡或一个进程里多跑几个 micro-batch 再更新，用时间换显存；数据并行是多张卡同时处理不同数据，用通信换吞吐。

**通信如何和计算 overlap？** backward 从后往前计算梯度，某些层梯度算完后可以立即启动 AllReduce，同时继续计算前面层梯度。DDP 的 bucket 机制就是为了这种重叠。

**checkpoint 要保存什么？** 至少保存 model state、optimizer state、scheduler state、step/epoch、随机种子和配置。分布式训练还要考虑 sharded checkpoint。

**多机训练失败怎么处理？** 需要定期 checkpoint、可恢复 dataloader state、任务重启机制、日志定位和失败节点隔离。大规模训练不能假设机器永不失败。

从 0 理解：

当模型或数据太大，单张 GPU 不够用，就要分布式训练。最容易理解的是数据并行：每张 GPU 放一份模型，但喂不同数据。每张卡算完梯度后，大家把梯度平均，再一起更新参数。

数据并行省的是训练时间，不省单卡模型显存；FSDP/ZeRO 则进一步把参数、梯度、优化器状态切开，解决单卡装不下的问题。Tensor Parallel 是把模型内部的大矩阵切开，适合超大模型；Pipeline Parallel 是把不同层放在不同设备上，像流水线一样执行。

面试表达：

“我会先判断瓶颈是模型装不下、batch 不够大、训练太慢还是通信太慢。不同问题对应不同并行策略：DDP 解决吞吐，FSDP/ZeRO 解决显存，Tensor/Pipeline Parallel 解决超大模型拆分。”

### 6.3 显存组成

训练显存：

- 参数。
- 梯度。
- 优化器状态。
- 激活值。
- 临时 workspace。
- 通信 buffer。

训练显存逐项解释：

- **参数**：模型权重本身，例如 Linear、Embedding、Norm 参数。
- **梯度**：每个可训练参数的梯度，反向传播后用于更新。
- **优化器状态**：Adam 会保存一阶矩、二阶矩，可能比参数还占显存。
- **激活值**：forward 中间结果，backward 要用。深模型和长序列时激活很大。
- **临时 workspace**：cuDNN/cuBLAS 或框架执行某些算子时申请的临时空间。
- **通信 buffer**：分布式训练中梯度 bucket、参数收集等通信临时缓冲。

推理显存：

- 模型权重。
- KV cache。
- activation 临时空间。
- batch 输入输出。
- runtime workspace。

推理显存逐项解释：

- **模型权重**：加载后的参数，占固定显存。
- **KV cache**：大模型自回归生成时缓存历史 K/V，随 batch、层数、头数和序列长度增长。
- **activation 临时空间**：当前 forward/decode 步的中间 tensor。
- **batch 输入输出**：输入 token、attention mask、输出 buffer。
- **runtime workspace**：推理框架为 kernel、通信、调度预留的工作区。

优化：

- mixed precision。
- activation checkpointing。
- ZeRO / FSDP。
- offload。
- quantization。
- sequence length 控制。
- micro-batch。

高频问题答案：

**训练显存为什么比推理大很多？** 训练要保存激活用于 backward，还要存梯度和优化器状态；推理只需要权重、临时激活和 KV cache。

**Activation checkpointing 怎么省显存？** 不保存所有中间激活，backward 时重新计算一部分 forward。它用额外计算换显存。

**ZeRO/FSDP 为什么省显存？** 它们把参数、梯度和优化器状态切分到多张卡上，每张卡只保存一部分，需要时再通信收集。

**量化为什么能省显存？** 用 int8/int4/fp8 等更低精度存权重或激活，减少内存占用和带宽压力，但可能带来精度损失。

从 0 理解：

显存不是只被模型权重占用。训练时还要存梯度、优化器状态和中间激活。Adam 类优化器通常还要为每个参数保存一阶矩和二阶矩，所以优化器状态可能比参数本身还占空间。

推理时不需要梯度和优化器状态，但大模型会有 KV cache。序列越长、batch 越大、层数越多、hidden size 越大，KV cache 越大。很多长上下文推理 OOM，真正原因不是权重，而是 KV cache。

可以这样估算思路：

```text
训练显存 ≈ 参数 + 梯度 + 优化器状态 + 激活 + 临时 buffer
推理显存 ≈ 权重 + KV cache + 临时 activation + runtime buffer
```

### 6.4 实验管理

必须记录：

- 代码版本。
- 数据版本。
- 配置。
- random seed。
- 训练日志。
- 指标。
- checkpoint。
- 环境依赖。

怎么做：

1. 每次实验固定一个配置文件，不手工散改参数。
2. 记录 git commit、数据版本、随机种子和环境。
3. 保存训练曲线、验证指标、失败样例和 checkpoint。
4. 对比实验只改一个关键变量，避免无法归因。
5. 重要实验写复盘：改了什么、为什么、结果如何、下一步做什么。

高频问题答案：

**为什么 random seed 还不一定完全复现？** GPU 并行、非确定性算子、DataLoader 多进程、库版本差异都可能导致细微差别。严格复现要设置 deterministic 选项，但可能牺牲性能。

**怎么做 ablation？** 一次只移除或替换一个组件，比较指标变化。例如去掉 rerank、换 loss、关闭增强，看每个设计是否真的贡献效果。

面试表达：

“实验不可复现通常比模型效果差更可怕，因为团队无法判断改动是否真的有效。我会把数据版本、代码 commit、配置和指标绑定记录。”

从 0 理解：

实验管理是为了避免“昨天好好的，今天不知道为什么变差”。如果你没有记录数据版本、代码版本、随机种子和配置，即使指标提升了，也很难证明是哪一个改动带来的。

一个合格实验记录至少包括：

```text
任务名、数据版本、代码 commit、模型结构、超参数、训练日志、验证指标、checkpoint、失败备注
```

导师面试中，这体现研究规范；工作面试中，这体现团队协作能力。

### 6.5 推理服务

核心指标：

- TTFT：time to first token。
- TPOT：time per output token。
- P50/P95/P99 latency。
- Throughput：requests/s 或 tokens/s。
- GPU utilization。
- Error rate。

关键技术：

- dynamic batching。
- continuous batching。
- KV cache。
- prefix caching。
- quantization。
- speculative decoding。
- streaming。
- request scheduling。
- backpressure。
- timeout and cancellation。

高频问题答案：

**TTFT 和 TPOT 区别是什么？** TTFT 是用户发请求到看到第一个 token 的时间，受排队、prefill 和调度影响；TPOT 是后续每个 token 的平均生成时间，反映 decode 效率。

**dynamic batching 解决什么？** 把多个请求合并成 batch，提高 GPU 利用率和吞吐。代价是可能增加等待时间，所以要受延迟 SLO 约束。

**continuous batching 为什么适合大模型？** 不同请求输出长度不同，传统 batch 要等最慢的请求结束。continuous batching 允许已完成请求退出、新请求加入，提高 GPU 利用率。

**backpressure 是什么？** 当请求量超过系统处理能力时，主动限流、排队或拒绝，避免队列无限增长导致全系统雪崩。

从 0 理解：

推理服务就是把用户请求送进模型，再把结果返回。普通 Web 服务主要处理数据库和业务逻辑；AI 推理服务还要处理 GPU 排队、batching、长输出、流式返回和显存管理。

大模型推理有两个阶段：

```text
prefill：处理输入 prompt，建立 KV cache
decode：一个 token 一个 token 生成输出
```

TTFT 主要受 prefill、排队和调度影响；TPOT 主要受 decode 效率影响。用户体感既关心第一个 token 多快出来，也关心后面生成是否流畅。

dynamic batching 把多个请求拼在一起跑，提高 GPU 利用率；continuous batching 允许请求生成过程中动态加入/退出，更适合输出长度不一致的大模型服务。

### 6.6 服务架构

常见部署：

```text
Client -> API Gateway -> Auth/RateLimit -> Inference Router -> Model Workers -> Metrics/Logs
```

要考虑：

- 多模型路由。
- 灰度发布。
- 版本回滚。
- 热更新。
- 资源隔离。
- 优先级队列。
- 限流和熔断。
- 监控和追踪。

高频问题答案：

**为什么需要灰度发布？** 新模型可能在某些场景退化。灰度先让少量流量使用新版本，观察指标稳定后再扩大，出问题可以快速回滚。

**多模型路由怎么做？** 可以按任务类型、用户等级、成本预算、输入长度和质量要求路由到不同模型。例如简单问题走小模型，复杂问题走大模型。

**限流和熔断区别是什么？** 限流是在入口控制请求量；熔断是在下游服务异常时暂时停止调用，快速失败或降级，避免故障扩散。

从 0 理解：

一个线上 AI 服务要防止“一个慢请求拖垮全系统”。所以需要网关、鉴权、限流、队列、worker、监控和降级策略。比如当 GPU 队列太长时，可以拒绝低优先级请求或切换小模型；当模型服务超时时，业务层要返回可解释的失败，而不是一直挂起。

典型请求路径：

```text
用户请求 -> 鉴权/限流 -> 参数校验 -> 任务排队 -> 模型 worker 推理 -> 流式/非流式返回 -> 日志和指标
```

系统设计面试中，要同时考虑正确性、延迟、吞吐、成本和故障恢复。

### 6.7 线上问题排查

延迟升高：

- batch 变大。
- 输入 token 变长。
- GPU 排队。
- KV cache 碎片。
- CPU tokenization 瓶颈。
- 网络或下游服务慢。

吞吐下降：

- GPU 利用率低。
- batch 策略不合适。
- kernel launch overhead。
- 数据拷贝过多。
- 模型 worker 重启。

结果变差：

- prompt 模板变更。
- 检索结果质量下降。
- 模型版本变化。
- 数据分布漂移。
- safety filter 误伤。

从 0 理解：

线上排查要先分类：是慢、错、贵，还是不稳定。不同问题看不同指标。延迟高看排队、batch、输入长度、下游依赖；结果差看模型版本、prompt、检索、数据分布；成本高看 GPU 利用率、输出长度、缓存命中率和模型路由。

一个可面试的排查框架：

```text
先确认现象 -> 看监控指标 -> 分层定位 -> 复现请求 -> 对比版本 -> 修复后压测/回归
```

不要只说“看日志”，要说看哪些日志、哪些指标、如何缩小范围。

高频问题答案：

**延迟突然升高怎么排查？** 先看请求量和输入输出 token 长度，再看队列长度、GPU 利用率、batch size、CPU 预处理、下游依赖和最近发布记录。

**回答质量突然变差怎么排查？** 对比模型版本、prompt 模板、检索结果、rerank 分数、知识库更新时间和 safety filter。RAG 场景要先看检索是否命中正确证据。

**成本突然升高怎么排查？** 看输出 token 是否变长、缓存命中率是否下降、是否更多请求路由到大模型、GPU 利用率是否低、重试率是否升高。

## 7. 大模型开发

大模型开发可以分成两类：一类是训练或微调模型本身，另一类是把大模型接入应用系统。0 基础先不要被 SFT、RLHF、DPO、RAG、Agent 这些词吓到，它们都围绕一个核心目标：**让模型在给定上下文和约束下，稳定地产生有用输出**。

大模型应用的基本框架：

```text
用户问题 -> 输入清洗/改写 -> 检索或工具调用 -> 组织 prompt -> 模型生成 -> 结构化解析/校验 -> 返回/记录反馈
```

大模型训练的基本框架：

```text
预训练学通用语言能力 -> 指令微调学会听指令 -> 偏好对齐学会更符合人类偏好 -> 领域适配学会专业任务
```

面试时要区分“模型能力问题”和“系统设计问题”。如果回答不准，可能是模型本身能力不够，也可能是 prompt 不清、检索证据错、工具返回错、上下文太长被截断、输出没有校验。

### 7.1 Transformer 语言模型

训练目标：

- 给定前文 token，预测下一个 token。
- 本质是大规模自监督学习。

Decoder-only 模型：

- GPT、LLaMA、Qwen 等。
- 用 causal mask 防止看到未来 token。
- 适合生成。

Encoder-only 模型：

- BERT 类。
- 适合理解、分类、抽取、embedding。

Encoder-decoder：

- T5、BART。
- 适合 seq2seq。

高频问题答案：

**Decoder-only 为什么适合生成？** 它用 causal mask 从左到右预测下一个 token，天然适合续写、对话、代码生成等任务。

**Encoder-only 为什么适合理解？** 它可以双向看完整输入，更适合分类、匹配、抽取和 embedding，但不天然适合自回归生成。

**为什么大模型训练目标是 next token prediction？** 文本天然有顺序，可以用海量无标注语料构造监督信号。模型通过预测下一个 token 学到语言结构和知识模式。

从 0 理解：

语言模型最基本的训练方式是预测下一个 token。例如输入“今天 天气 很”，模型应该预测“好”的概率更高。只要语料足够大，模型会在这个任务中学到语法、知识、推理模式和常见表达。

Decoder-only 模型适合生成，因为它从左到右一个 token 一个 token 续写。Causal mask 的作用是防止模型训练时偷看未来答案。Encoder-only 模型更适合理解任务，比如分类、检索 embedding；Encoder-decoder 模型适合把一个序列转换成另一个序列，比如翻译和摘要。

面试表达：

“大语言模型的基础目标是 next token prediction。Decoder-only 架构通过 causal attention 做自回归生成，推理时每一步基于已有上下文生成下一个 token，因此 KV cache 和解码调度对性能很关键。”

### 7.2 Tokenizer

需要掌握：

- BPE、WordPiece、SentencePiece。
- special tokens。
- token 数和字符数不同。
- 中文、代码、数字、空格会影响 tokenization。

常见坑：

- 截断导致关键信息丢失。
- prompt 模板中空格和特殊符号影响输出。
- 多语言 token 效率差异。

高频问题答案：

**token 和字符有什么区别？** token 是 tokenizer 切分后的模型输入单位，可能是一个字、一个词、一个子词或符号片段。一个汉字不一定等于一个 token，一个英文单词也可能被拆成多个 token。

**为什么 tokenizer 会影响成本和速度？** 上下文长度、计费、KV cache、prefill 计算都按 token 增长。token 越多，推理越慢、显存越高、成本越大。

**长文档怎么处理？** 不能直接硬塞。要分块、检索、摘要或层级组织，确保关键内容进入上下文，并记录被截断内容的风险。

从 0 理解：

模型不直接看汉字或英文单词，而是看 token。Tokenizer 负责把文本切成 token id。不同 tokenizer 切法不同，同一句中文、英文、代码可能 token 数差异很大。

为什么 token 重要？因为大模型的上下文长度按 token 计算，计费也常按 token 计算，推理速度和 KV cache 大小也和 token 数有关。一个看似短的输入，如果包含大量代码、表格或特殊符号，可能 token 很多。

工程上要注意：长文档进模型前要分块；关键内容不能被截断；prompt 模板改了一个空格，有时也可能影响输出格式。

### 7.3 位置编码

常见方法：

- Sinusoidal positional encoding。
- Learned position embedding。
- RoPE。
- ALiBi。

RoPE 直觉：

- 把位置信息编码进 Q/K 的旋转变换。
- 相对位置信息更自然。
- 长上下文扩展常围绕 RoPE scaling 做文章。

高频问题答案：

**为什么 Transformer 需要位置编码？** Self-attention 本身对输入顺序不敏感，如果没有位置编码，模型很难区分“我喜欢你”和“你喜欢我”的顺序差异。

**RoPE 和绝对位置编码区别是什么？** 绝对位置编码直接给每个位置一个向量；RoPE 把位置信息融入 Q/K 的旋转关系，更自然地表达相对位置，现代大模型常用。

**长上下文为什么和位置编码有关？** 如果模型训练时只见过 4K 长度，推理扩到 32K，位置编码外推可能不稳定，需要 RoPE scaling、插值或继续训练。

从 0 理解：

Transformer 本身看一组 token 时，并不知道 token 的先后顺序。如果没有位置编码，“我喜欢你”和“你喜欢我”可能只是一堆 token 集合，顺序信息不明确。位置编码就是告诉模型每个 token 在序列中的位置。

RoPE 常用于现代大模型，它把位置信息融合到 attention 的 Q/K 中，让模型更自然地感知相对位置。长上下文扩展时，位置编码是否还能在超出训练长度后稳定工作，是重要问题。

### 7.4 预训练、微调和对齐

阶段：

- Pretraining：大规模无标注语料，学通用能力。
- SFT：高质量指令数据，学会按指令回答。
- Preference learning：RLHF、DPO、IPO 等，让输出更符合偏好。
- Domain adaptation：领域继续训练或 LoRA 微调。

这些阶段具体怎么做：

- **Pretraining 怎么做**：收集大规模文本，清洗去重，按 next token prediction 训练模型。目标不是完成某个具体任务，而是学通用语言、知识和模式。
- **SFT 怎么做**：准备 instruction-response 数据，例如“用户问题 -> 理想回答”，用监督学习让模型学会按指令输出。
- **Preference learning 怎么做**：准备偏好数据，例如同一问题下答案 A 比答案 B 好。RLHF 用奖励模型和强化学习，DPO 直接用偏好对优化模型。
- **Domain adaptation 怎么做**：用领域文本继续训练，或用 LoRA 在领域问答/报告数据上微调，让模型适应医学、法律、代码等特定任务。

LoRA：

- 冻结原模型参数。
- 在某些线性层旁加低秩矩阵。
- 训练参数少，显存需求低。
- 适合领域适配和风格适配。

LoRA 怎么实际接入：

1. 选择注入层，常见是 attention 的 `q_proj/v_proj` 或 FFN 线性层。
2. 冻结原模型权重，只训练 LoRA 的低秩矩阵 A 和 B。
3. 训练完成后可以单独保存 adapter，也可以 merge 回原权重。
4. 多个任务可以保存多个 adapter，根据场景加载。

QLoRA：

- 基座模型量化加载。
- LoRA 参数保持可训练。
- 更省显存。

高频问题答案：

**预训练、SFT、RLHF/DPO 区别是什么？** 预训练让模型通过海量文本学语言和知识；SFT 用指令问答数据教模型按人类指令回答；RLHF/DPO 用偏好数据让模型更符合人类偏好，例如更有帮助、更安全、更少胡说。

**LoRA 为什么省参数？** LoRA 冻结原始大矩阵，只训练两个低秩小矩阵。原来要更新 `d x d` 的参数，现在只更新 `d x r + r x d`，其中 `r` 很小，所以训练参数和显存都少。

**什么时候用微调，什么时候用 RAG？** 如果要改变回答风格、格式、任务习惯，用微调；如果要接入经常变化或需要引用的事实知识，用 RAG；如果要查询实时数据或执行业务动作，用工具调用。

**QLoRA 的核心是什么？** 用低比特量化加载基座模型，减少显存占用，同时只训练 LoRA adapter。它适合资源有限时做领域适配。

从 0 理解：

预训练像让模型读了大量书，学到语言和世界知识；SFT 像给模型看“用户这样问，你应该这样答”的示范；偏好学习像告诉模型“两个答案里哪个更好”；领域微调像让模型补专业课。

LoRA 的直觉是：不直接改动原来巨大的权重矩阵，而是在旁边加一个低秩更新。这样训练参数少、显存省、可以快速适配领域。它适合让模型学格式、风格、领域表达，但不一定能可靠注入大量事实知识。事实知识经常更适合 RAG 或数据库工具。

面试中要避免说“微调能解决一切”。微调用于改变模型行为模式；RAG 用于接入外部知识；工具调用用于获取实时或结构化结果。

### 7.5 RAG

基本链路：

```text
Query -> Rewrite -> Retrieve -> Rerank -> Compose Prompt -> Generate -> Cite/Verify
```

模块：

- 文档解析。
- chunking。
- embedding。
- vector store。
- hybrid search。
- reranker。
- prompt assembly。
- answer generation。
- citation verification。

RAG 模块逐项解释：

- **文档解析**：把 PDF、网页、Word、Markdown、表格等转成干净文本。解析失败会让后面检索全错。
- **chunking**：把长文档切成小块。切块要尽量保持语义完整，例如按标题、段落、条款切。
- **embedding**：把 query 和 chunk 转成向量，使语义相近的文本在向量空间里更接近。
- **vector store**：存储向量并支持近邻检索，例如 FAISS、Milvus、pgvector。
- **hybrid search**：同时用向量检索和关键词检索。关键词适合精确术语，向量适合语义相似。
- **reranker**：对召回候选做更精细排序。它通常慢一些，但能显著提高 top 结果质量。
- **prompt assembly**：把用户问题、检索证据、回答约束、引用格式拼成模型输入。
- **answer generation**：模型基于证据生成答案。要约束“只根据证据回答”。
- **citation verification**：检查答案中的引用是否真的支持结论，减少幻觉。

关键问题：

- chunk 多大？
- overlap 多少？
- embedding 模型选哪个？
- 如何处理表格、图片、PDF？
- 如何避免检索无关内容？
- 如何判断回答是否忠实于证据？

怎么做：

1. 解析文档：把 PDF、网页、Markdown、表格转成可检索文本。
2. 切 chunk：按语义段落或标题切分，必要时加 overlap。
3. 向量化：用 embedding 模型把 chunk 变成向量。
4. 检索：根据 query 找 top-k chunk，可结合关键词检索。
5. rerank：用更强模型重新排序，保留最相关证据。
6. 组 prompt：把问题、证据、约束和输出格式放入上下文。
7. 生成与校验：要求引用证据，检查回答是否越界或幻觉。

高频问题答案：

**chunk 多大合适？** 太小会丢上下文，太大检索不准且浪费 token。一般从 300 到 800 中文字或按标题段落开始实验，再用 retrieval recall 和答案质量调。

**overlap 有什么用？** 防止答案所需信息被切在两个 chunk 边界。overlap 太少会断语义，太多会重复、增加索引和上下文噪声。

**embedding 和 reranker 区别是什么？** embedding 用向量快速召回候选，速度快但可能粗；reranker 对 query-chunk 做更精细匹配，速度慢但排序更准。

**如何判断 RAG 幻觉？** 看回答中的关键事实是否被检索证据支持；检查引用是否真实命中；遇到证据不足时是否拒答，而不是补编。

面试回答：

“RAG 的瓶颈往往不是生成模型，而是检索和证据组织。我的评估会拆成 retrieval recall、rerank precision、答案事实性、引用覆盖率和端到端任务成功率。”

从 0 理解：

RAG 是“先查资料，再回答”。大模型本身的参数知识可能过期、缺失或无法追溯。RAG 把外部文档检索出来放进上下文，让模型基于证据回答。

RAG 常见失败点：

- 文档解析失败：PDF、表格、图片没有正确变成文本。
- chunk 切得不好：一个答案需要的信息被切散。
- embedding 找错：语义相似但事实不相关。
- rerank 不准：真正有用的证据排在后面。
- prompt 塞太多：上下文噪声干扰模型。
- 生成不忠实：模型没有严格依据证据。

一个可面试的 RAG 系统要分别评估检索和生成。只看最终回答，很难知道问题出在检索还是模型。

### 7.6 Agent 和工具调用

Agent 能力：

- 规划。
- 调用工具。
- 读取工具结果。
- 继续推理。
- 完成任务。

Agent 能力逐项解释：

- **规划是什么**：把一个复杂任务拆成多个步骤，例如先查病历、再查影像结果、再生成报告。
- **调用工具怎么做**：模型输出结构化工具名和参数，系统校验后执行真实函数或 API。
- **读取工具结果怎么做**：工具返回 JSON、文本或表格后，模型把结果纳入上下文继续判断。
- **继续推理是什么**：模型根据工具结果决定下一步，可能继续调用工具，也可能结束回答。
- **完成任务是什么**：输出最终结果，或触发业务动作。高风险动作必须有权限和人审。

工程风险：

- 工具参数错误。
- 无限循环。
- 幻觉工具结果。
- 权限越界。
- 成本不可控。
- 结果不可复现。

治理：

- 明确工具 schema。
- 工具调用前权限检查。
- 最大步数限制。
- 日志审计。
- 对关键动作做人审。
- 用 deterministic workflow 替代不必要的自由规划。

怎么做：

1. 明确工具列表和 schema：每个工具的参数、返回值、权限。
2. 让模型只生成结构化工具调用，不直接执行危险动作。
3. 工具执行前做参数校验、权限校验和额度校验。
4. 工具返回后让模型基于真实结果继续回答。
5. 设置最大轮数、超时、失败兜底和日志审计。

高频问题答案：

**Agent 和普通 Chatbot 区别是什么？** 普通 Chatbot 主要生成文本；Agent 可以根据任务调用工具、读取结果并继续决策。Agent 更强，但风险和不可控性也更高。

**为什么 Agent 容易不稳定？** 模型可能选错工具、填错参数、误解工具返回、循环调用或越权。工程上要靠 schema、权限、步数限制和确定性流程约束。

**什么时候不该用 Agent？** 如果流程固定、风险高、规则明确，优先用普通后端流程或状态机，只在需要语言理解和灵活决策处使用模型。

从 0 理解：

Agent 可以理解成“模型不只回答，还能决定调用哪些工具”。例如用户问“帮我查这个患者最近一次检查结果并生成报告”，模型可能需要调用数据库、影像服务、报告服务。Agent 的价值在于连接工具，风险也在于工具调用会影响真实系统。

工程上不要迷信“让模型自己规划一切”。越关键的业务，越要把流程做成可控状态机：哪些工具能调用、参数是什么、失败怎么处理、是否需要人确认，都要明确。模型适合做语言理解和灵活决策，但权限、安全和最终动作要由系统约束。

### 7.7 大模型推理优化

KV cache：

- 自回归生成中，历史 token 的 K/V 可以缓存。
- 避免每步重复计算全部历史。
- 长上下文和大 batch 时 KV cache 显存很关键。

KV cache 怎么计算直觉：

每一层 attention 都会为历史 token 保存 K 和 V。序列越长、batch 越大、层数越多、head 越多，KV cache 越大。推理 OOM 很多时候不是权重装不下，而是长上下文和并发请求的 KV cache 装不下。

Paged attention：

- 把 KV cache 管理成分页块。
- 降低碎片，提高服务吞吐。

Paged attention 怎么做：

它把每个请求的 KV cache 拆成固定大小的 block/page，像操作系统管理内存一样按页分配。这样不同长度请求不会造成大量连续显存浪费，也更适合 continuous batching。

Continuous batching：

- 请求可以动态加入和退出 batch。
- 适合生成长度不同的在线请求。

Continuous batching 怎么做：

传统 batching 要等一个 batch 中所有请求都结束；continuous batching 每个 decode step 都可以把已完成请求移出，把新请求加入。这样 GPU 不会因为某些短请求结束而空出位置。

Speculative decoding：

- 小模型先草拟多个 token。
- 大模型验证。
- 如果命中率高，可以降低解码延迟。

Speculative decoding 怎么做：

先用小模型一次生成几个候选 token，再让大模型并行验证这些 token 是否可接受。接受越多，越能减少大模型逐 token 解码次数；如果小模型质量差，收益就小。

量化：

- 权重量化：INT8、INT4。
- 激活量化。
- KV cache 量化。
- FP8。

量化类型逐项解释：

- **权重量化**：把模型权重从 fp16/bf16 压到 int8/int4，主要减少模型显存和加载带宽。
- **激活量化**：推理中间激活也用低精度表示，难度更高，因为激活分布随输入变化。
- **KV cache 量化**：把缓存的 K/V 降低精度，适合长上下文和高并发，但要评估生成质量。
- **FP8**：更低精度浮点，依赖硬件支持，训练和推理都可能用，但数值稳定性要仔细处理。

高频问题答案：

**KV cache 为什么能加速？** 自回归生成时历史 token 的 K/V 不变，可以缓存起来。下一步只算新 token 的 Q/K/V，并和历史 K/V 交互，避免重复计算整个前缀。

**Paged attention 解决什么？** 多请求生成长度不同，KV cache 容易碎片化。Paged attention 把 KV cache 分页管理，提高显存利用率和调度灵活性。

**Speculative decoding 什么时候有效？** 小模型草拟 token，大模型验证。如果小模型命中率高，就能一次接受多个 token，减少大模型调用步数；如果命中率低，收益会下降。

**量化为什么不一定更快？** 低精度减少显存和带宽，但如果硬件不支持高效低精度计算，或需要频繁 dequantize，速度可能不升反降。还要评估精度损失。

注意：

- 量化不是免费午餐，要评估精度、吞吐、延迟和硬件支持。
- 不同模型、任务、硬件上的收益差异很大。

从 0 理解：

大模型推理慢，主要因为模型大、序列长、生成是逐 token 的。训练时可以并行处理整段文本；生成时第 101 个 token 依赖前 100 个 token，所以 decode 阶段天然有串行性。

KV cache 的作用是避免每一步重算历史。Paged attention 的作用是更高效地管理很多请求的 KV cache。Continuous batching 的作用是让不同长度请求更充分利用 GPU。Speculative decoding 的作用是用小模型先猜，大模型批量验证，从而减少大模型解码步数。

量化通过低精度表示权重或激活，减少显存和带宽压力。但量化可能损失精度，尤其在数学、代码、长上下文或领域任务上要单独评估。

### 7.8 大模型评测

离线评测：

- 通用能力 benchmark。
- 领域题库。
- 人工标注测试集。
- 对抗样例。
- 安全测试。

离线评测怎么做：

- **通用 benchmark**：快速了解模型基础能力，但不能代表业务表现。
- **领域题库**：针对医学、金融、代码等业务知识构造测试集。
- **人工标注测试集**：由人标出标准答案、证据和评分规则，是最可靠的业务评测之一。
- **对抗样例**：故意构造诱导、边界、冲突输入，测试鲁棒性。
- **安全测试**：测试隐私泄露、越权、危险建议、prompt injection 等风险。

在线评测：

- A/B test。
- 用户满意度。
- 任务完成率。
- 反馈采样审核。
- 延迟和成本。

在线评测怎么做：

- **A/B test**：小流量对比新旧模型，观察质量和系统指标。
- **用户满意度**：点赞、差评、人工反馈、投诉等信号，但要注意噪声。
- **任务完成率**：用户目标是否完成，比“回答像不像”更接近业务价值。
- **反馈采样审核**：抽样看真实对话，分析错误类型。
- **延迟和成本**：模型再好，如果太慢太贵，也可能无法上线。

RAG 评测：

- 检索召回。
- 证据相关性。
- 回答忠实性。
- 引用准确率。
- 拒答正确率。

高频问题答案：

**为什么大模型评测不能只看通用 benchmark？** 通用 benchmark 不能代表你的业务数据、prompt、工具、RAG 知识库和延迟成本。上线前必须做领域测试集和真实任务评估。

**RAG 评测怎么拆？** 先评估检索是否找到正确证据，再评估 rerank 是否把正确证据排前面，最后评估生成是否忠实引用证据。

**Agent 怎么评测？** 看任务完成率、工具选择准确率、参数正确率、失败恢复能力、越权率、成本和平均步数。

**人工评测怎么设计？** 给评审标准，例如事实正确、表达清晰、引用充分、安全合规、任务完成。最好做双人标注和一致性检查。

从 0 理解：

大模型评测不能只问“回答看起来好不好”。要拆成多个维度：是否完成任务，事实是否正确，格式是否符合要求，是否引用证据，遇到不知道的问题是否拒答，是否安全，延迟和成本是否可接受。

离线评测用于上线前筛选版本；在线评测用于观察真实用户场景。RAG 应用还要单独评估 retrieval，因为检索错了，模型再强也可能答错。Agent 应用要评估工具调用是否正确、是否越权、是否能从工具错误中恢复。

面试表达：

“我会把大模型评测拆成任务成功率、事实性、格式遵循、鲁棒性、安全性和系统指标。对于 RAG，再拆 retrieval recall、rerank precision 和答案忠实性；对于 Agent，再看工具调用准确率和失败恢复。”

## 8. 典型项目怎么准备

项目是面试里最能拉开差距的部分。0 基础准备项目，不要只准备“我用了什么技术”，而要准备“我为什么这样做、遇到什么问题、如何验证有效、如果继续做会怎么改”。

项目讲述的通用框架：

```text
背景和目标 -> 数据和输入输出 -> 方法和架构 -> 训练/实现细节 -> 评估指标 -> 遇到的问题 -> 改进方向
```

面试官通常会沿着项目追问基础知识。例如你说用了 U-Net，他可能问 Dice Loss；你说用了 RAG，他可能问 chunk 和 rerank；你说用了 CUDA，他可能问 shared memory；你说上线服务，他可能问 P95 延迟和降级。所以项目准备不是背稿，而是把项目和知识体系连起来。

### 8.1 医学影像 AI 项目讲法

推荐结构：

1. 背景：头颅 CT / 病灶检测 / 金属伪影 / 报告辅助。
2. 数据：DICOM、标注、预处理、训练验证切分。
3. 模型：U-Net、CNN、Transformer 或检测模型。
4. 训练：loss、增强、指标、调参。
5. 部署：FastAPI、异步任务、模型服务、报告服务。
6. 评估：Dice、AUC、医生审核、一致性。
7. 风险：误诊风险、人审闭环、数据隐私。
8. 改进：真实 checkpoint、跨院泛化、主动学习、RAG 报告增强。

一句话版本：

“我做的是一个面向头部 CT 的 AI 辅助流程，不只是训练模型，而是把影像上传、模型推理、结果解释、报告生成、医生审核和平台业务流串成闭环。技术上涉及医学影像预处理、深度模型推理、RAG 知识增强、微服务接口和用户级验收。”

从 0 理解：

医学影像项目的关键是“高风险场景下的可信辅助”。模型不能替代医生，也不能只报一个分数就结束。你要讲清楚影像怎么预处理，模型输出如何解释，报告如何生成，医生如何审核，错误如何兜底。

建议准备 3 个具体例子：

- 一个模型识别正确的例子：说明流程有效。
- 一个模型识别错误的例子：说明你做过误差分析。
- 一个工程链路问题：说明你理解真实系统不是只跑模型。

这样导师会看到研究意识，企业会看到工程闭环。

高频追问答案：

**DICOM 预处理怎么做？** 先读取像素和元信息，做 HU 值转换、窗宽窗位、spacing 统一、方向校正、尺寸裁剪或重采样，再归一化成模型输入。训练和推理必须用同一套预处理。

**医学影像怎么避免数据泄漏？** 按患者而不是按 slice 切分；同一患者所有序列只能出现在同一个集合；还要检查 DICOM 元信息、文件名、设备信息是否泄漏标签。

**分割模型怎么评估？** 常用 Dice、IoU、Hausdorff distance，也要看小病灶召回和边界质量。高风险场景不能只看平均 Dice，要做病例级错误分析。

**模型错了怎么办？** 不能直接把错误结果给临床使用。应输出置信度、证据或热力图，进入医生审核流程；错误样本进入反馈池，用于后续再训练或规则兜底。

### 8.2 RAG 项目讲法

推荐结构：

1. 文档来源和数据治理。
2. chunk 策略。
3. embedding 和向量库。
4. rerank。
5. prompt 组织。
6. 引用和事实性校验。
7. 线上反馈和知识库更新。

不要只说“我接了向量数据库”。要讲：

- 为什么这么切 chunk。
- 检索失败时怎么发现。
- 如何评价答案是否基于证据。
- 如何处理过期知识和冲突知识。

从 0 理解：

RAG 项目最容易被问穿，因为很多人只会“embedding + vector db + prompt”。真正可面试的 RAG 要能讲清每一步为什么存在。chunk 是为了把长文档变成可检索单元；embedding 是为了语义召回；rerank 是为了在召回结果中重新排序；prompt 是为了把证据组织给模型；引用校验是为了降低幻觉。

准备回答时，最好用一个真实 query 贯穿：

```text
用户问了什么 -> 系统检索到哪些 chunk -> rerank 后保留哪些 -> prompt 如何拼接 -> 模型如何回答 -> 如何判断引用正确
```

高频追问答案：

**为什么不用直接把所有文档塞进 prompt？** 上下文长度有限，成本和延迟高，且无关内容会干扰模型。RAG 先检索相关证据，减少噪声和 token 成本。

**检索失败怎么发现？** 构造带标准答案和标准证据的评测集，统计 top-k retrieval recall；线上记录 query、召回 chunk、最终答案和用户反馈，抽样审查。

**过期知识怎么处理？** 给文档加版本、时间戳和来源优先级；检索时优先新版本；生成时要求引用来源；知识更新后重建或增量更新索引。

**冲突知识怎么处理？** 不让模型自行编结论，而是展示冲突来源、时间和适用条件；必要时拒答或转人工审核。

### 8.3 算子项目讲法

推荐结构：

1. 选一个常见算子：matmul、softmax、layernorm、topk、embedding lookup。
2. Python baseline。
3. CUDA/Triton 实现。
4. 正确性验证。
5. benchmark。
6. profiler 分析。
7. 优化迭代。
8. 和 PyTorch / cuBLAS / FlashAttention 类库比较。

面试亮点：

- 你能解释为什么快，而不是只贴一个速度数字。
- 你能解释什么情况下不快。
- 你能解释精度误差和边界条件。

从 0 理解：

算子项目的价值不在于“我写过 CUDA 文件”，而在于你理解性能优化方法论。一个好项目应该包含 baseline、优化版本和对比。比如 softmax 项目可以先用 PyTorch reference，再写 naive CUDA，再写 fused/reduction 优化版本，最后比较不同 batch、行长度、dtype 下的性能。

准备材料时一定要有表格：

```text
实现方式 | 输入规模 | 平均耗时 | 加速比 | 最大误差 | 主要瓶颈
```

有这个表，面试官会觉得你真的测过，而不是只看过教程。

高频追问答案：

**为什么要先写 Python reference？** reference 保证语义正确。CUDA/Triton 优化前必须知道正确输出，否则速度再快也没有意义。

**怎么证明优化真的有效？** 同一输入规模下对比 baseline、优化版和框架原生实现；报告 warmup 后平均耗时、加速比、误差、显存和适用 shape。

**为什么某些 shape 下不快？** 可能 block size 不合适、访存不连续、occupancy 下降、Tensor Core 对齐不好、kernel launch overhead 占比高。要按 shape 分析，不要只报一个数字。

**精度误差怎么处理？** fp16/bf16 下允许一定误差，用 `max_abs_error`、`max_rel_error` 或 `torch.allclose` 判断；reduction 顺序不同也会造成微小差异。

### 8.4 大模型应用项目讲法

推荐结构：

1. 任务：问答、报告生成、客服、代码、数据分析。
2. 模型选择：闭源 API、本地开源模型、混合路由。
3. Prompt：模板、变量、约束、输出 schema。
4. RAG：检索和证据。
5. 工具调用：数据库、搜索、业务 API。
6. 服务：流式输出、超时、重试、缓存。
7. 评测：自动评测 + 人审。
8. 安全：隐私、越权、幻觉、注入攻击。

从 0 理解：

大模型应用项目的难点是可靠性。Demo 很容易，稳定上线很难。你要讲清楚模型什么时候回答、什么时候检索、什么时候调用工具、什么时候拒答、输出如何校验、失败如何处理。

一个强回答可以这样组织：

```text
我没有只依赖模型自由生成，而是把流程拆成检索、生成、校验和记录反馈。对结构化输出使用 schema 校验；对事实性回答要求引用证据；对高风险动作加入权限和人审；对线上效果通过任务成功率、引用准确率和用户反馈评估。
```

高频追问答案：

**为什么要结构化输出？** 业务系统需要稳定解析模型结果。用 JSON schema、字段校验和重试修复可以减少格式漂移。

**如何防止 prompt injection？** 不让用户输入覆盖系统指令；工具调用做权限校验；RAG 文档内容只作为证据，不作为系统命令；对敏感动作做人审。

**如何处理模型幻觉？** 对事实性回答要求引用证据；证据不足时拒答；关键答案做规则或二次校验；线上抽样审查并沉淀失败样例。

**如何评估大模型应用项目？** 不只看“回答好不好”，还要看任务成功率、引用准确率、格式正确率、拒答正确率、延迟、成本和用户反馈。

## 10. 面试题清单

面试题不要用“背答案”的方式准备。更好的方式是给每道题建立一个回答模板：

```text
先给定义 -> 解释为什么需要它 -> 讲一个例子 -> 说常见坑或工程取舍
```

例如“BatchNorm 和 LayerNorm 区别”：

```text
定义：BatchNorm 沿 batch 统计，LayerNorm 沿特征维统计。
为什么：它们都用于稳定训练，但适用数据和架构不同。
例子：CNN 常用 BatchNorm，Transformer 常用 LayerNorm。
坑：小 batch 时 BatchNorm 统计不稳定；推理时 BatchNorm 使用 running stats。
```

### 10.1 机器学习

**交叉熵和 KL 散度关系？** 交叉熵 `H(p, q) = H(p) + KL(p || q)`。真实分布 `p` 固定时，最小化交叉熵等价于最小化 KL 散度，也就是让预测分布 `q` 接近真实分布。

**AUC 如何计算？** AUC 可以理解为随机抽一个正样本和一个负样本，模型把正样本排在负样本前面的概率。它看排序能力，不依赖固定阈值。

**类别不平衡怎么办？** 数据侧重采样或补充少数类；loss 侧加权或 focal loss；指标侧看 PR-AUC、recall、precision 而不是只看 accuracy；上线侧根据误报漏报代价调阈值。

**训练集和测试集分布不一致怎么办？** 先确认切分方式是否合理，再做分桶评估找出差异来源。可用领域适配、重新采样、收集目标域数据、数据增强或上线后持续监控。

**GBDT 和随机森林区别？** 随机森林是 bagging，并行训练多棵树再平均，主要降低方差；GBDT 是 boosting，串行训练，每棵树拟合前面模型的残差，主要降低偏差。

**为什么需要验证集？** 验证集用于调参、选模型和 early stopping。测试集应留到最后评估泛化能力，不能反复用来调参。

**如何判断数据泄漏？** 检查同一用户/患者/时间未来信息是否进入训练；检查特征中是否包含标签派生字段；检查训练和测试是否共享重复样本或高度相似样本。

### 10.2 深度学习

**BatchNorm 和 LayerNorm 区别？** BatchNorm 在 batch 维度统计均值方差，适合 CNN，但小 batch 不稳定；LayerNorm 在单个样本特征维统计，不依赖 batch size，适合 Transformer。

**Dropout 训练和推理有什么区别？** 训练时随机置零部分神经元，降低过拟合；推理时关闭随机置零，使用完整网络，输出更稳定。

**Adam 和 AdamW 区别？** Adam 使用一阶、二阶矩自适应更新；AdamW 把 weight decay 从梯度更新中解耦，避免 L2 正则和自适应学习率耦合，大模型更常用。

**梯度消失和梯度爆炸怎么处理？** 梯度消失可用 ReLU、残差、归一化、合适初始化；梯度爆炸可用 gradient clipping、降低学习率、检查 loss 和混合精度溢出。

**Transformer 为什么能并行？** 训练时 self-attention 可以一次性计算所有 token 之间的关系，不像 RNN 必须按时间步串行。因此更适合 GPU 并行。

**Attention 复杂度是多少？** 标准 self-attention 对序列长度是 `O(n^2)`，因为每个 token 要和其他 token 计算相似度。长上下文需要稀疏、分块、KV cache 等优化。

**为什么要 residual？** 残差连接让层学习 `F(x)+x`，信息和梯度可以更容易穿过深层网络，缓解深层退化和训练困难。

### 10.3 PyTorch

**`model.train()` 和 `model.eval()` 影响什么？** 主要影响 Dropout 和 BatchNorm。训练模式启用随机 dropout 和 batch 统计；评估模式关闭 dropout，并使用 BatchNorm 的 running stats。

**`torch.no_grad()` 和 `inference_mode()` 区别？** 二者都关闭梯度，减少显存；`inference_mode()` 更激进，禁用 autograd 版本计数等机制，推理更快，但不适合后续还要参与梯度计算的 tensor。

**`detach()` 的作用？** 从当前计算图中分离 tensor，后续操作不会把梯度传回原图。常用于日志、缓存、停止梯度。

**为什么会 OOM？** 可能是 batch 太大、输入序列太长、模型/activation 太大、保存了计算图、没用 `no_grad()`、临时 tensor 过多。先看显存组成再处理。

**DataLoader 变慢怎么排查？** 分别统计 data time 和 compute time；检查磁盘 IO、解码、增强、`num_workers`、`pin_memory`、小文件数量和 CPU 利用率。

**如何做混合精度训练？** 用 `autocast` 让部分算子低精度运行，用 `GradScaler` 防止 fp16 梯度下溢。bf16 场景通常更稳定，可能不需要 scaler。

**如何保存和加载 checkpoint？** 保存 model、optimizer、scheduler、epoch/step、配置和随机种子。恢复时先创建同结构对象，再 load state dict，保证 device 和 dtype 正确。

### 10.4 CUDA / 算子

**thread、block、grid、warp 的关系？** Grid 是一次 kernel 的所有 block；block 包含多个 thread；warp 通常是 32 个 thread，是实际调度单位。block 内线程可共享 shared memory。

**shared memory 什么时候有用？** 当一小块数据会被同一个 block 内多个线程重复使用时有用，例如 tiled matmul。若数据只用一次，同步和搬运成本可能超过收益。

**什么是 coalesced access？** 相邻线程访问相邻内存地址，GPU 可以合并访存事务，提高带宽利用率。

**如何优化 reduction？** 先让每个 block 内用 shared memory 或 warp primitive 做局部 reduce，再合并 block 结果，尽量减少全局 atomic 热点，并注意数值误差。

**softmax 怎么保证数值稳定？** 每行先减最大值，再做 `exp` 和归一化。减最大值不改变结果，但避免指数溢出。

**kernel fusion 为什么减少显存访问？** 多个算子拆开会把中间结果写回 global memory，再读出来。fusion 让中间值留在 register/shared memory，减少读写和 launch overhead。

**如何 benchmark CUDA kernel？** 先 warmup，再用 CUDA event 计时，计时前后同步；比较不同 shape/dtype，报告误差、耗时和加速比。

**Nsight Systems 和 Nsight Compute 各看什么？** Systems 看端到端时间线和 CPU/GPU/通信重叠；Compute 看单 kernel 的 occupancy、访存、stall、Tensor Core 利用率。

### 10.5 AI Infra

**DDP 如何同步梯度？** 每张卡本地 forward/backward，梯度产生后通过 AllReduce 求平均，保证所有卡用一致梯度更新参数。

**ZeRO/FSDP 为什么省显存？** 它们把参数、梯度、优化器状态切分到多张卡，每张卡只保留一部分，需要完整参数时再 AllGather。

**checkpoint 保存哪些内容？** 模型参数、优化器状态、scheduler、训练步数、配置、随机种子，分布式训练还要保存分片信息和恢复策略。

**推理服务如何做动态 batching？** 请求进入队列后，在短时间窗口内合并成 batch 送入 GPU。要限制最大等待时间和最大 batch，避免延迟超标。

**如何处理超时和取消？** 请求层设置 deadline，队列中可取消，执行中尽量支持中断或忽略返回；下游超时要快速失败、降级或重试，避免资源泄漏。

**如何做模型灰度发布？** 小比例流量先走新模型，监控质量、延迟、错误率和成本；指标稳定再扩大，异常时快速回滚。

**如何设计监控指标？** 系统指标看 QPS、P50/P95/P99、错误率、GPU 利用率；模型指标看任务成功率、拒答率、幻觉率、引用准确率；业务指标看转化、节省时间或满意度。

### 10.6 大模型

**Decoder-only 和 Encoder-only 区别？** Decoder-only 用 causal mask 自回归生成，适合对话和生成；Encoder-only 双向理解输入，适合分类、匹配、抽取和 embedding。

**RoPE 是什么？** RoPE 是旋转位置编码，把位置信息融入 Q/K 的旋转关系中，能表达相对位置信息，现代大模型常用。

**LoRA 为什么省参数？** 冻结原模型，只训练低秩增量矩阵，参数量从完整大矩阵降到两个小矩阵，显存和训练成本都低。

**RAG 为什么会幻觉？** 可能是检索没找到正确证据、rerank 排错、prompt 证据太噪、模型没有严格遵循证据，或问题本身证据不足却没有拒答。

**KV cache 为什么能加速？** 自回归生成时历史 token 的 K/V 不变，缓存后每步不用重算全部历史，只处理新 token。

**speculative decoding 的适用条件？** 小模型草拟结果和大模型最终结果一致率较高时有效；如果小模型经常猜错，验证开销会抵消收益。

**量化会带来什么风险？** 可能损失精度，影响数学、代码、长上下文和领域任务；也可能因硬件或 dequantize 开销导致速度收益不明显。

**如何评估大模型应用？** 拆成任务成功率、事实性、格式遵循、安全性、鲁棒性、延迟和成本。RAG 还要评估检索召回、证据相关性和引用准确率。

练习方法：

1. 每个小节挑 5 道题，先不看资料，用自己的话回答。
2. 回答录音，检查是否有“只背名词、不讲逻辑”的问题。
3. 每道题补一个项目例子。
4. 对不会的问题标记为“概念不会、公式不会、工程不会、表达不会”中的一种。

面试时如果不会，可以这样处理：

“这个细节我没有完整实现过，但我会从输入输出、计算瓶颈和评估方式来分析。我的理解是……如果要验证，我会先做一个 baseline，再用 profiler/实验指标确认。”

## 11. 高频回答模板

模板不是让你机械背诵，而是让你的回答有结构。面试中最怕两种回答：一种是只说结论，另一种是发散太久没有重点。推荐每个回答控制在：

```text
一句话结论 -> 2 到 3 个关键点 -> 一个项目例子 -> 一个改进方向
```

### 11.1 “你觉得你做的项目难点是什么？”

模板：

“难点不只是模型训练，而是数据、模型和系统之间的一致性。比如数据侧要保证切分不泄漏、标签可信；模型侧要选择合适指标并做误差分析；系统侧要把推理结果接入业务流程，并考虑延迟、失败兜底和人工审核。我当时主要解决了 A、B、C，后来发现 D 还可以继续改进。”

### 11.2 “模型效果不好怎么办？”

回答顺序：

1. 先确认评估是否可信：数据切分、指标、标签质量。
2. 看训练是否正常：loss、过拟合/欠拟合、学习率。
3. 做误差分析：哪些类别、场景、样本失败。
4. 改数据：清洗、补标、增强、重采样。
5. 改模型：结构、loss、预训练、微调。
6. 改部署：阈值、规则兜底、人审。

### 11.3 “为什么不用更大的模型？”

回答：

“更大的模型可能提升上限，但也增加延迟、显存、成本和部署复杂度。我会先用简单 baseline 明确任务是否可学，再看误差来自容量不足、数据不足还是分布问题。如果瓶颈确实是模型表达能力，再考虑更大模型或预训练模型。”

### 11.4 “如何做一个高性能推理服务？”

回答：

“我会从请求画像开始：输入长度、输出长度、并发、延迟 SLO、成本约束。系统上做动态 batching、流式输出、缓存、限流、超时取消和监控。模型侧考虑量化、KV cache、prefix caching、speculative decoding。最后用 P50/P95/P99、tokens/s、GPU 利用率和错误率做压测，而不是只看单请求速度。”

### 11.5 “如何优化一个 CUDA 算子？”

回答：

“先写正确的 baseline，再用 CUDA event 和 profiler 建立性能基线。然后判断是 memory-bound 还是 compute-bound。memory-bound 就减少 global memory 访问、改善 coalescing、做 fusion；compute-bound 就提高并行度、使用 Tensor Core 或优化指令流水。每次只改一个点，用 benchmark 验证，避免凭感觉优化。”

如何把模板变成自己的话：

- 把“项目”替换成你真实做过的模块。
- 把“指标”替换成你真实记录过的数字。
- 把“困难”替换成你真实踩过的坑。
- 把“改进”替换成你下一步能做的小实验。

如果没有真实数字，也可以诚实说：“当前我做的是功能闭环和本地验证，下一步会补压测和定量指标。”诚实比编造数字更稳。

## 12. 导师面试研究方向表达

导师面试的表达重点和求职不同。求职强调“我能解决问题”，导师面试还要强调“我能提出问题、验证假设、持续深入”。不要只说“我想研究大模型”，要说清楚你观察到什么问题、为什么重要、可能怎么做。

推荐结构：

```text
我感兴趣的方向 -> 我做过的相关项目 -> 我看到的不足 -> 我想进一步研究的问题 -> 可能方法和验证方式
```

### 12.1 如果你想做医学 AI

可以这样讲：

“我对医学影像 AI 和临床辅助系统比较感兴趣。相比单纯追求模型指标，我更关心模型如何进入真实流程：数据质量、跨设备泛化、可解释性、医生审核闭环、报告生成和安全边界。我希望研究方向能结合深度学习、医学影像和可信 AI，例如弱监督分割、多模态报告生成、RAG 辅助诊断或模型不确定性估计。”

可延展问题：

- 如何用少量标注训练医学模型？
- 如何评估模型跨医院泛化？
- 如何让报告生成有证据可追溯？
- 如何把医生反馈变成持续学习数据？

### 12.2 如果你想做 AI Infra

可以这样讲：

“我对 AI 系统方向感兴趣，尤其是训练和推理效率。大模型能力越来越强，但真实落地受限于显存、通信、延迟和成本。我希望研究高效推理、分布式训练、算子优化或模型服务调度，例如 KV cache 管理、continuous batching、混合并行、通信计算重叠等问题。”

可延展问题：

- 如何提升多用户在线推理吞吐？
- 如何降低长上下文推理显存？
- 如何让训练在多机故障下更稳定？
- 如何为特定模型定制高性能算子？

### 12.3 如果你想做大模型应用

可以这样讲：

“我对大模型应用和智能体系统感兴趣，但我认为重点不是简单调用 API，而是如何让模型在真实业务中可靠完成任务。这里涉及 RAG、工具调用、结构化输出、权限控制、评测体系和线上反馈。我希望研究如何提升模型回答的事实性、可控性和可评估性。”

从 0 准备导师面试时，要把“我会哪些技术”升级成“我想研究哪些问题”。例如：

- 技术说法：我会 RAG。
- 研究说法：我关注 RAG 在专业领域中的证据忠实性和冲突知识处理。
- 技术说法：我会医学影像分割。
- 研究说法：我关注小样本标注条件下的医学影像泛化和医生反馈闭环。
- 技术说法：我会 CUDA。
- 研究说法：我关注大模型推理中 attention、KV cache 和算子融合带来的端到端效率优化。

## 13. 简历 bullet 写法

简历 bullet 的目标是让面试官愿意追问你，并且你能接得住。不要堆技术名词，要写清楚你在什么背景下做了什么、解决了什么、怎么验证。

弱写法：

- 使用深度学习完成 CT 图像识别。
- 使用大模型完成问答系统。
- 使用 CUDA 优化算法。

强写法：

- 构建头部 CT AI 辅助分析流程，覆盖 DICOM 预处理、模型推理、报告生成和医生审核闭环；通过端到端验收脚本验证上传、分析、报告和回写链路。
- 设计医学知识 RAG 模块，完成知识入库、向量检索、rerank、证据拼接和安全表达规则，提升报告生成的可追溯性。
- 实现 CUDA/Triton softmax 或 layernorm 算子，对齐 PyTorch reference，并使用 CUDA event / profiler 分析 kernel launch、访存和 reduction 瓶颈。
- 搭建大模型流式推理服务，支持 SSE 输出、超时控制、错误兜底、日志追踪和 P95 延迟监控。

简历 bullet 公式：

```text
做了什么 + 用了什么技术 + 解决什么问题 + 如何验证 + 结果或影响
```

从 0 改简历的方法：

1. 先写事实：我做了哪个模块。
2. 补技术：用了哪些框架、模型、系统组件。
3. 补问题：为什么要做，解决什么痛点。
4. 补验证：测试、指标、验收、压测、人工审核。
5. 补影响：提升、闭环、稳定性、可维护性。

如果没有量化指标，也可以写“通过端到端验收脚本验证”“完成真实业务链路闭环”“支持错误兜底和人工审核”。但后续最好补上指标，如 P95 延迟、准确率、Dice、吞吐或测试覆盖。

## 14. 最小作品集建议

如果要同时覆盖导师面试和找工作，建议准备 4 个小作品：

1. ML baseline：表格分类或医学影像分类，包含数据划分、指标和误差分析。
2. DL 项目：CNN/U-Net/Transformer 训练，包含训练曲线和 ablation。
3. AI Infra 项目：模型推理服务，包含 batching、streaming、metrics。
4. 算子项目：CUDA/Triton kernel，包含 correctness、benchmark、profile。

每个作品都要有：

- README。
- 运行命令。
- 数据说明。
- 指标结果。
- 失败分析。
- 下一步改进。

从 0 执行作品集时，规模要小，但闭环要完整。一个能跑通、能解释、能复现实验的小项目，比一个宏大但跑不起来的项目更有价值。

每个作品都要回答：

```text
别人怎么运行？
输入输出是什么？
核心技术点是什么？
怎么证明它是对的？
它现在的不足是什么？
下一步怎么改？
```

作品集最后可以形成一个统一叙事：你不是零散学了 ML、DL、CUDA、大模型，而是在构建“从模型训练到系统部署再到性能优化”的完整能力。

## 16. 最后复习清单

面试前一天确认自己能讲清：

- 一个 ML 问题从数据到指标怎么定义。
- 一个 DL 模型从 forward 到 backward 怎么训练。
- 一个 Transformer block 的 shape 和复杂度。
- 一个 PyTorch OOM 如何排查。
- 一个 CUDA kernel 的线程映射和访存优化。
- 一个分布式训练任务的通信模式。
- 一个大模型推理服务的延迟和吞吐优化。
- 一个 RAG 系统的检索、生成和事实性评估。
- 一个项目的失败案例和改进方向。

最重要的心态：

- 不要装作什么都懂。
- 会的讲机制，不会的讲排查路径。
- 不只讲模型，也讲数据、系统、评估和风险。
- 对导师展示研究问题意识；对企业展示解决真实问题的闭环能力。

最后一天不要再大面积学新内容，而是做三件事：

1. 用 3 分钟讲清自己的主项目。
2. 用 30 秒回答每个高频基础题。
3. 把不会的问题整理成“我会如何进一步验证”的排查路径。

面试官通常不怕你有不会的点，怕的是你不会拆问题。只要你能把问题拆成数据、模型、训练、系统、性能、评估几个层次，并说明下一步怎么验证，就已经具备从 0 走到可面试的核心能力。
