# 病灶识别模型接入开发指导

## 1. 阶段目标

当前 `HeadCTOrchestrator` 已完成：

```text
主平台 -> Orchestrator -> Filter -> Orchestrator 汇总
```

下一阶段考虑接入病灶识别模型，但不应把病灶识别代码直接塞进 Orchestrator。

推荐目标是：

```text
Orchestrator 作为统一编排入口
LesionDetection 作为独立病灶识别服务
Filter 继续负责影像质控与金属伪影识别
```

目标调用链：

```text
主平台
  -> HeadCTOrchestrator
    -> Filter
    -> LesionDetection
    -> 汇总 quality_control + lesion_analysis + report_assist
```

## 2. 模块边界

### 2.1 Orchestrator 负责什么

Orchestrator 负责：

- 接收主平台请求。
- 创建统一任务 ID。
- 调用 Filter 获取质控结果。
- 根据质控结果判断是否适合继续病灶识别。
- 调用病灶识别服务。
- 汇总结构化结果。
- 生成报告辅助字段。
- 统一返回给主平台。

Orchestrator 不负责：

- 训练病灶识别模型。
- 直接实现具体模型推理细节。
- 直接输出最终医学诊断。

### 2.2 LesionDetection 负责什么

LesionDetection 负责：

- 加载病灶识别模型。
- 接收 CT 输入。
- 完成单病种或多病种识别。
- 输出结构化病灶结果。
- 输出必要的 mask、bbox、preview 或置信度。

LesionDetection 不负责：

- 调用 Filter。
- 汇总报告。
- 管理主平台任务。
- 生成最终诊断结论。

## 4. 第一阶段建议：先做病灶识别服务契约

第一阶段不要求马上训练或接入真实模型。

建议先实现：

```text
HeadCTLesionDetection 合同服务 / mock 服务
```

用途：

- 固定接口。
- 固定 JSON 字段。
- 让 Orchestrator 可以先接入并编排。
- 后续真实模型只需要替换推理内部实现。

第一阶段接口：

```text
GET  /api/head-ct-lesion/health
POST /api/head-ct-lesion/tasks
GET  /api/head-ct-lesion/tasks/{task_id}
GET  /api/head-ct-lesion/results/{task_id}
```

任务状态：

```text
queued
running
success
failed
```

## 5. LesionDetection 输入契约

### 5.1 上传文件方式

第一阶段建议支持：

```http
POST /api/head-ct-lesion/tasks
Content-Type: multipart/form-data
```

字段：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `file` | 是 | `.nii` 或 `.nii.gz` CT 文件 |
| `case_context` | 否 | 平台上下文 JSON 字符串 |
| `quality_context` | 否 | Filter 质控结果 JSON 字符串 |
| `requested_lesions` | 否 | 逗号分隔，如 `hemorrhage,fracture` |

### 5.2 为什么要传 quality_context

Orchestrator 已经调用过 Filter，LesionDetection 不应重复调用 Filter。

`quality_context` 用于告知病灶模型：

- 是否存在金属伪影。
- 伪影严重程度。
- 受影响层面。
- 是否应降低结果可信度。

示例：

```json
{
  "artifact_detected": true,
  "artifact_ratio": 0.034,
  "severity": "moderate",
  "affected_slices": [42, 43, 44],
  "analysis_reliability": "limited_by_artifact"
}
```

## 7. 单个病灶结果字段说明

| 字段 | 说明 |
| --- | --- |
| `lesion_type` | 机器可读病灶类型 |
| `display_name` | 中文显示名 |
| `detected` | 是否检出 |
| `confidence` | 置信度，0 到 1 |
| `severity` | `none` / `suspected` / `mild` / `moderate` / `severe` |
| `affected_slices` | 受影响层面 |
| `locations` | 解剖位置，第一阶段可为空 |
| `bbox` | 检测框，第一阶段可为空 |
| `mask_url` | 分割 mask 下载地址，可为空 |
| `preview_urls` | 预览图地址 |
| `model_name` | 模型名称 |
| `model_version` | 模型版本 |
| `reliability` | 可信度状态 |
| `warnings` | 模型和质控提示 |
| `report_suggestion` | 报告辅助提示，不是最终诊断 |

## 8. 病灶类型规划

建议分阶段接入。

### 第一优先级：颅内出血

原因：

- 临床急诊价值高。
- 数据集相对更常见。
- 分类任务可先行，后续再扩展定位/分割。

建议模型阶段：

```text
阶段 1：Study / Series 级二分类
阶段 2：slice 级定位
阶段 3：出血区域分割
阶段 4：出血亚型分类
```

输出：

```json
{
  "lesion_type": "intracranial_hemorrhage",
  "detected": true,
  "confidence": 0.91,
  "affected_slices": [30, 31, 32],
  "report_suggestion": "疑似颅内出血，请医生结合原始影像复核。"
}
```

### 第二优先级：颅骨骨折

特点：

- 更依赖骨窗。
- 可能需要不同预处理窗宽窗位。
- 可先做 slice 级或 series 级分类。

### 第三优先级：中线移位 / 占位效应

特点：

- 更偏结构测量。
- 可能需要脑实质分割或关键线检测。
- 不建议作为第一个模型。

### 第四优先级：脑梗死

特点：

- 早期 CT 表现隐匿。
- 对数据质量和标注要求高。
- 不建议在没有可靠标注前优先做。

## 9. Orchestrator 接入 LesionDetection 的方式

Orchestrator 当前结果中已有：

```json
{
  "lesion_analysis": {
    "status": "not_configured",
    "enabled": false,
    "results": [],
    "warnings": []
  }
}
```

接入后改为：

```json
{
  "lesion_analysis": {
    "status": "success",
    "enabled": true,
    "task_id": "lesion_task_xxx",
    "result_url": "http://localhost:8020/api/head-ct-lesion/results/lesion_task_xxx",
    "results": [],
    "warnings": []
  }
}
```

Orchestrator 调用链：

```text
1. 创建 Orchestrator 任务
2. 调用 Filter
3. 获取 Filter result
4. 根据 Filter severity 决定是否调用 LesionDetection
5. 调用 LesionDetection /tasks
6. 轮询 LesionDetection /tasks/{task_id}
7. 获取 LesionDetection /results/{task_id}
8. 合并 lesion_analysis
9. 合并 report_assist
```

## 10. 根据质控结果决定是否调用病灶模型

建议策略：

```text
severity == none:
  正常调用病灶识别

severity == mild:
  正常调用病灶识别，附加轻度伪影 warning

severity == moderate:
  调用病灶识别，但在结果中标注 limited_by_artifact

severity == severe:
  默认仍可调用，但结果标注 strongly_limited_by_artifact
  如果模型未通过强伪影数据验证，可以配置为跳过病灶识别
```

配置项建议：

```text
LESION_SKIP_ON_SEVERE_ARTIFACT=false
LESION_REQUESTED_TYPES=hemorrhage
```

## 11. 报告辅助合并规则

Orchestrator 应合并：

```text
Filter report_suggestion
LesionDetection report_suggestion
安全免责声明
```

输出示例：

```json
{
  "report_assist": {
    "summary": "存在中度金属伪影，可能影响邻近区域判断。未见明确颅内出血征象，建议医生结合原始影像复核。",
    "quality_control_text": "存在中度金属伪影，可能影响邻近区域判断。",
    "lesion_text": "未见明确颅内出血征象。",
    "warnings": [
      "AI 结果仅供辅助参考，最终结论需医生审核。",
      "金属伪影可能影响局部病灶识别结果。"
    ],
    "can_enter_report": true
  }
}
```

`can_enter_report` 仅表示可以作为草稿素材进入医生审核，不表示自动形成最终诊断。

## 13. Smoke Test 规划

### LesionDetection 服务测试

至少覆盖：

```text
GET  /api/head-ct-lesion/health
POST /api/head-ct-lesion/tasks
GET  /api/head-ct-lesion/tasks/{task_id}
GET  /api/head-ct-lesion/results/{task_id}
```

测试场景：

- 正常 `.nii.gz` 输入。
- 非法文件类型。
- 不支持的病灶类型。
- 任务不存在。
- 模型不可用。

### Orchestrator 集成测试

至少覆盖：

- Filter 成功 + LesionDetection 成功。
- Filter 成功 + LesionDetection 不可用。
- Filter 严重伪影 + LesionDetection 跳过。
- LesionDetection 返回阳性结果。
- LesionDetection 返回阴性结果。

## 14. 第一阶段 Mock LesionDetection 建议

为了先打通架构，可以先做 mock 版本：

```text
HeadCTLesionDetection/mock mode
```

行为：

- 接收 CT。
- 不加载真实模型。
- 返回固定阴性结果。
- 根据 `quality_context.severity` 添加 warning。

示例：

```json
{
  "lesion_type": "intracranial_hemorrhage",
  "detected": false,
  "confidence": 0.05,
  "severity": "none",
  "report_suggestion": "未见明确颅内出血征象，建议医生结合原始影像复核。"
}
```

这样可以先验证：

```text
主平台 -> Orchestrator -> Filter -> LesionDetection -> Orchestrator
```

再替换真实模型。

## 15. 真实模型接入建议

真实模型接入时，每个模型至少需要：

- 训练脚本。
- 推理脚本。
- 模型结构文件。
- 权重路径配置。
- 输入预处理说明。
- 输出后处理说明。
- 验证集指标。
- smoke sample。
- 模型版本号。

模型目录示例：

```text
HeadCTLesionDetection/models/hemorrhage/
  model.py
  infer.py
  train.py
  config.py
  README.md
  runs/
    hemorrhage_v1/
      best.pt
      metrics.json
```

## 16. 数据集建议

颅内出血可考虑使用已有 CQ500 相关数据作为第一阶段研究数据。

注意：

- CQ500 更适合分类标签研究。
- 若要做精确分割，需要额外 mask 标注。
- 训练前需要明确标签字段含义。
- 不要把金属伪影 mask 当成病灶标签。

第一阶段建议从：

```text
series/study 级分类
```

开始，不要一开始就做病灶分割。
