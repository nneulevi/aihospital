# 前端模块衔接与验收指导

## 1. 当前结论

当前 `frontend` 已具备与 `Project2` 主平台衔接的基础能力：项目使用 Vue 3 + Vite + TypeScript，已通过 Orval 从 `http://localhost:8092/v3/api-docs` 生成 API 客户端，并提供患者端、医生端、管理员端三类页面。

当前 P0 衔接项已完成：患者病历详情、管理员 AI 排班、财务收费/退费/记录/日结、药房库存/发药/退药均已移除模拟 API，并改为调用 Project2 真实接口。医生接诊页也已补充 AI 影像上传、影像识别和 AI 报告生成入口。后续验收重点从“补接口”转为“在真实 Project2 数据和账号下逐项验证业务落库与结果展示”。

## 4. 后端代理与接口生成

`vite.config.ts` 已将本地 `/api` 代理到 Project2：

```ts
server: {
  port: 5173,
  proxy: {
    '/api': {
      target: 'http://localhost:8092',
      changeOrigin: true,
    }
  }
}
```

`orval.config.mjs` 当前从 Project2 获取 OpenAPI：

```js
input: { target: 'http://localhost:8092/v3/api-docs' }
```

当后端接口变化时，建议重新生成 API 客户端：

```powershell
cd D:\exam\frontend
npx orval --config orval.config.mjs
```

注意：`src/api/client.ts` 和 `src/api/model/` 是生成物，不应直接手改；业务页面应优先通过 `src/api/index.ts` 暴露的包装函数调用接口。

## 7. 开闭原则落地方式

后续补齐时应尽量保持现有结构稳定：

1. 不直接修改 Orval 生成的 `client.ts` 和 `model/` 类型文件。
2. 新接口优先在 `src/api/index.ts` 做一层语义化封装。
3. 页面只替换模拟 API 的调用点，不重写页面布局。
4. 如果接口返回字段与页面字段不一致，优先新增局部 adapter 函数，而不是改后端实体或生成类型。
5. 通用错误处理继续放在 `src/utils/request.ts`，页面只处理业务提示。
6. 新增页面时只扩展 `router/index.ts`，不破坏现有患者端、医生端、管理员端路由分组。

## 9. 与头部 CT AI 子系统的衔接点

前端不直接调用 `HeadCTOrchestrator`、`Filter`、`HeadCTLesionDetection`、`HeadCTReportService` 等 Python 服务，而是通过 Project2 的 AI API 间接接入。

推荐链路：

```text
frontend
  -> Project2 /api/ai/*
  -> HeadCTOrchestrator
  -> Filter / Lesion / RAG / Report / EMR
```

因此前端验收时只需要确认：

1. `Project2` 的 `/api/ai/consultation/triage`、`/api/ai/diagnosis/suggest` 能返回真实结果。
2. 影像上传和分析接口返回任务或分析结果。
3. AI报告生成接口返回可审核、可写入 EMR 的结构化报告。
4. 页面不展示 Python 子服务内部字段，只展示 Project2 对前端稳定暴露的 VO。

5. 若后端 OpenAPI 变化，先执行 `npx orval --config orval.config.mjs`，再做页面适配和 `npm run build`。
